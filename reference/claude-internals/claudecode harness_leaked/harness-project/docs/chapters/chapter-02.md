# Chapter 2: 3-에이전트 아키텍처

## 2.1 데이터 흐름 개관

3-에이전트 하네스의 데이터 흐름은 단방향 파이프라인과 순환 피드백 루프의 조합이다.

```
사용자 프롬프트 ("Build a DAW")
    │
    ▼
┌─────────┐    ProductSpec     ┌──────────────┐
│ Planner │ ─────────────────► │              │
└─────────┘                    │              │
                               │ Orchestrator │
       ┌──────────────────────►│              │
       │                       │              │
       │  Handoff              └──────┬───────┘
       │                              │
       │                              │ ProductSpec + Handoff (or ∅)
       │                              ▼
       │                       ┌─────────────┐
       │                       │  Generator   │ ──► 앱 코드
       │                       └─────────────┘
       │                              │
       │                              │ 빌드 완료
       │                              ▼
       │                       ┌─────────────┐
       └───────────────────────│  Evaluator   │ ──► Evaluation
                (FAIL/PARTIAL) └─────────────┘
                                      │
                                      │ PASS
                                      ▼
                               cost-report.json
                               + 완성된 앱
```

이 흐름에서 단방향 구간과 순환 구간을 구분해야 한다.

**단방향 구간**: 사용자 프롬프트 → Planner → ProductSpec. Planner는 1회만 실행되며, 생성된 스펙은 전체 실행 동안 불변이다. Planner의 비용이 전체의 0.4%에 불과한 이유가 여기에 있다.

**순환 구간**: Generator → Evaluator → Handoff → Generator. 이 루프가 하네스의 핵심 메커니즘이다. Evaluator의 verdict가 PASS가 아닌 한, Generator는 Handoff에 기반하여 수정 빌드를 반복한다.

데이터 흐름의 모든 접점에서 교환되는 데이터는 고정 스키마로 정의된다. ProductSpec, Evaluation, Handoff, AcceptanceContract의 4가지 스키마가 에이전트 간 계약(contract)으로 기능한다. 이 스키마의 설계는 Chapter 7에서 상세히 다룬다.

## 2.2 각 에이전트의 역할과 경계

### 2.2.1 Planner: 프롬프트에서 스펙으로

Planner의 입력은 사용자의 자연어 프롬프트(1-4문장)이며, 출력은 ProductSpec 스키마를 따르는 구조화된 제품 스펙이다.

Planner의 역할 경계는 명확하다:

**Planner가 결정하는 것:**
- 기능 목록과 우선순위
- 디자인 언어 (색상 팔레트, 타이포그래피, 레이아웃 원칙)
- 고수준 기술 아키텍처
- AI 통합 방안

**Planner가 결정하지 않는 것:**
- 세부 구현 방식 (어떤 라이브러리의 어떤 API를 사용할지)
- 파일 구조
- 데이터베이스 스키마 상세
- API 엔드포인트 설계

이 경계 설정의 근거를 PDF에서 찾을 수 있다:

> "If the planner tried to specify granular technical details upfront and got something wrong, the errors in the spec would cascade."

Planner가 세부 기술 결정까지 내리면, 그 결정의 오류가 Generator에 전파된다. Generator는 스펙을 충실히 이행하도록 설계되었으므로, 잘못된 기술 결정도 그대로 구현한다. 이 전파 효과를 차단하려면 Planner의 결정 범위를 "무엇을" 수준으로 제한하고, "어떻게"는 Generator에 위임해야 한다.

실제 산출물의 예시. 사용자가 "DAW 앱을 만들어라"라고 입력하면, Planner는 다음과 같은 스펙을 생성한다:

- **기능 16개**: 멀티트랙 타임라인, 오디오 녹음, 미디 편집, 이펙트 체인, 믹서 콘솔, 프로젝트 저장/로드, 실시간 파형 표시, BPM/박자 설정, 트랙 뮤트/솔로, 볼륨 자동화, 플러그인 시스템, 키보드 단축키, 되돌리기/다시하기, 내보내기(WAV/MP3), 협업 공유, AI 작곡 보조
- **디자인 언어**: 어두운 테마, 네온 강조색, 모노스페이스 계기판, 최소주의 아이콘
- **AI 통합**: AI 작곡 보조 (멜로디 생성, 코드 진행 제안, 자동 마스터링)

1문장이 16기능 스펙으로 확장되었다. 이것이 Planner의 범위 확장 효과다.

### 2.2.2 Generator: 스펙에서 실행 가능한 앱으로

Generator는 하네스에서 가장 비용이 큰 에이전트다. 전체 비용의 91.3%를 차지한다. 그 이유는 Generator가 실제로 파일을 생성하고, 코드를 작성하고, 빌드를 실행하는 유일한 에이전트이기 때문이다.

Generator의 동작은 라운드에 따라 분기한다.

**Round 1 (초기 빌드):**
- 입력: ProductSpec + 스택 프리셋 + 디자인 스킬
- 행동: 스펙의 모든 기능을 처음부터 구현
- 비용: ~$71 (PDF 기준)

**Round N (수정 빌드):**
- 입력: ProductSpec + Handoff (이전 라운드 피드백)
- 행동: Handoff의 strategic_direction에 따라 REFINE 또는 PIVOT
- 비용: Round 2 ~$37, Round 3 ~$6 (급격히 감소)

Generator에게 허용된 도구는 읽기와 쓰기 모두를 포함한다: 파일 읽기, 파일 쓰기, 셸 실행, 패턴 검색. 이는 Planner(읽기 전용)와 Evaluator(읽기 전용 + Playwright)와 대비되는 권한 구성이다.

Generator의 비용 감소 패턴은 하네스 비용 구조의 핵심 특성이다. Round 1에서 $71이던 비용이 Round 3에서 $6으로 떨어지는 이유는, 수정 범위가 라운드마다 축소되기 때문이다. Round 1은 전체 앱을 구축하지만, Round 2는 Evaluator가 지적한 항목만 수정하고, Round 3는 더 적은 항목만 수정한다. 이 수렴 특성이 하네스의 비용 예측 가능성을 보장한다.

### 2.2.3 Evaluator: 독립 평가와 피드백 생성

Evaluator는 Generator의 산출물을 평가하되, Generator의 컨텍스트와 완전히 격리된 상태에서 평가한다. 이 격리가 하네스 아키텍처의 가장 중요한 설계 결정이다.

Evaluator의 평가 축은 4가지다:

| 평가 축 | 가중치 | 평가 내용 |
|---------|--------|----------|
| design_quality | 높음 | 디자인이 개별 부품의 집합이 아닌 통일된 전체로 느껴지는가 |
| originality | 높음 | 템플릿 레이아웃, 라이브러리 기본값, AI 생성 패턴의 흔적이 있는가 |
| craft | 보통 | 타이포그래피 위계, 간격 일관성, 색상 조화, 대비율 |
| functionality | 보통 | 인터페이스의 기능을 사용자가 이해하고 작업을 완료할 수 있는가 |

각 축은 1-10 점수로 채점되며, 축별 임계값(threshold) 미달 시 verdict가 FAIL이 된다. 어떤 축이든 하나라도 임계값 미만이면 전체가 FAIL이다. 총점 평균이 아닌 개별 축 기준의 hard threshold를 적용하는 이유는, 총점 방식이 심각한 결함을 다른 축의 높은 점수로 상쇄하는 것을 허용하기 때문이다.

Evaluator는 verdict 외에 두 가지 추가 산출물을 생성한다:

1. **Findings**: 구체적 발견 사항 목록. 각 항목은 severity(critical/high/medium), category(design/functionality/performance/accessibility), description, location, evidence를 포함한다.
2. **Strategic Direction**: REFINE 또는 PIVOT. 현재 방향을 유지하며 수정할 것인지(REFINE), 미학적 접근 전체를 재설계할 것인지(PIVOT)를 결정한다.

## 2.3 GAN에서 영감받은 피드백 루프

### 2.3.1 Generator-Discriminator 비유

하네스의 Generator-Evaluator 루프는 GAN(Generative Adversarial Network)의 Generator-Discriminator 구조와 표면적으로 유사하다.

GAN에서:
- Generator는 실제 데이터와 구별 불가능한 합성 데이터를 생성하려 한다.
- Discriminator는 실제 데이터와 합성 데이터를 구별하려 한다.
- 두 네트워크의 적대적 경쟁이 양쪽의 성능을 동시에 향상시킨다.

하네스에서:
- Generator는 스펙을 만족하는 앱을 생성하려 한다.
- Evaluator는 스펙 대비 산출물의 결함을 식별하려 한다.
- 두 에이전트의 반복이 산출물 품질을 향상시킨다.

### 2.3.2 비유의 한계

이 비유에는 중요한 한계가 있다.

**적대적이 아닌 협력적 관계.** GAN의 Generator와 Discriminator는 서로의 손실 함수를 최대화/최소화하는 적대적 관계다. 하네스의 Generator와 Evaluator는 동일한 목표(스펙 만족)를 공유하는 협력적 관계다. Evaluator가 Generator를 "이기려는" 것이 아니라, Generator가 더 나은 산출물을 생성하도록 구체적 피드백을 제공한다.

**학습이 아닌 지시.** GAN에서 Generator는 Discriminator의 gradient를 통해 파라미터를 갱신한다. 하네스의 Generator는 Evaluator의 피드백을 자연어 지시로 수신하여 다음 빌드에 반영한다. 파라미터 갱신이 아닌 프롬프트 갱신이다.

**수렴 보장의 부재.** GAN은 Nash 균형으로의 수렴이 이론적으로 분석 가능하다(실제로는 불안정한 경우가 많지만). 하네스의 루프에는 그러한 수렴 보장이 없다. 따라서 외부 종료 조건(최대 라운드 수, 예산 한도, 점수 정체 감지)이 필수적이다.

이 한계에도 불구하고 GAN 비유가 유용한 이유는, "생성과 평가의 분리"와 "반복적 품질 향상"이라는 두 핵심 원리를 직관적으로 전달하기 때문이다. 하네스 설계자는 이 비유를 출발점으로 삼되, 위의 차이점을 명확히 인식해야 한다.

## 2.4 비용 구조 분석

### 2.4.1 비용 분포

PDF V2 하네스의 비용 분포는 극도로 비대칭적이다:

| 구성요소 | 비용 | 비중 |
|---------|------|------|
| Planner | $0.46 | 0.4% |
| Generator (3라운드 합산) | $113.85 | 91.3% |
| Evaluator (3라운드 합산) | $10.39 | 8.3% |
| **합계** | **$124.70** | **100%** |

이 분포에서 두 가지 통찰을 도출할 수 있다.

### 2.4.2 오케스트레이션 오버헤드의 미미함

Planner와 Evaluator의 합산 비용은 $10.85로, 전체의 8.7%다. 이 8.7%가 하네스 아키텍처가 추가하는 순수 오버헤드다. 나머지 91.3%는 Generator, 즉 Solo 실행에서도 발생하는 코드 생성 비용이다.

Solo 실행의 비용이 $9이고 하네스의 Generator 비용이 $113.85인 이유는, 하네스의 Generator가 훨씬 넓은 범위(16개 기능)를 구현하기 때문이다. 동일 범위를 Solo로 구현한다면 Solo의 비용도 유사한 수준으로 증가할 것이다. 즉, 비용 차이의 대부분은 하네스 오버헤드가 아니라 범위 확장의 결과다.

### 2.4.3 라운드별 비용의 급격한 감소

| 라운드 | Generator | Evaluator | 합계 |
|--------|-----------|-----------|------|
| Round 1 | $71.08 (57.0%) | $3.24 (2.6%) | $74.32 |
| Round 2 | $36.89 (29.6%) | $3.09 (2.5%) | $39.98 |
| Round 3 | $5.88 (4.7%) | $4.06 (3.2%) | $9.94 |

Generator의 비용이 Round 1에서 Round 3으로 $71.08 → $36.89 → $5.88로 감소한다. 감소율은 각각 48%, 84%다. 이 급격한 감소는 각 라운드에서 Generator의 수정 범위가 축소되기 때문이다.

반면 Evaluator의 비용은 $3.24 → $3.09 → $4.06으로 거의 일정하다. Evaluator는 매 라운드 전체 앱을 평가해야 하므로, 수정 범위와 무관하게 일정한 비용이 발생한다. Round 3에서 약간 증가한 것은 앱의 복잡도가 누적되어 평가 대상이 증가했기 때문으로 추정된다.

이 비대칭적 비용 구조는 하네스의 비용 효율성을 결정하는 핵심 요인이다. 추가 라운드의 한계 비용은 주로 Generator 비용으로 구성되며, 그 Generator 비용은 라운드마다 급감한다. 3라운드 이후의 4번째 라운드 비용은 $5-10 수준으로 예상되며, 이는 추가 라운드의 비용 부담이 초기 라운드 대비 극히 낮음을 의미한다.

그러나 비용이 낮다고 해서 추가 라운드가 항상 유효한 것은 아니다. 점수 정체(plateau)가 발생하면 추가 비용이 품질 개선 없이 소비된다. 이것이 Orchestrator의 plateau detection이 필요한 이유다. 비용 제어와 품질 수렴의 관계는 Chapter 11에서 상세히 다룬다.
