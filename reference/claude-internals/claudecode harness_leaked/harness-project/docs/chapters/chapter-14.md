# Chapter 14: 모델 진화와 하네스 적응

## 14.1 모델 능력 향상이 하네스에 미치는 영향

하네스의 모든 구성요소는 특정 시점의 모델 능력을 전제로 설계되었다. 모델이 진화하면 이 전제가 변동한다. V1→V2 진화에서 이미 관찰된 현상이다.

### 14.1.1 스프린트 구조 제거 사례

V1(Opus 4.5)에서 V2(Opus 4.6)로의 전환에서 가장 극적인 변화는 스프린트 구조의 제거였다. 이 사례가 하네스 적응의 원형(archetype)을 제공한다.

**V1의 전제**: Opus 4.5는 장기 일관성이 부족하여, 대규모 빌드를 소규모 스프린트로 분해해야 한다.

**V2의 현실**: Opus 4.6의 장기 일관성이 향상되어, 스프린트 분해 없이도 대규모 빌드가 일관적으로 수행된다.

**적응 결과**: 스프린트 구조 제거. 비용 $200 → $124.70 (37.7% 감소). 실행 시간 6시간 → 3시간 50분 (36.1% 감소).

이 사례에서 추출할 수 있는 일반 원칙: 구성요소 제거는 비용 감소와 복잡도 감소를 동시에 달성한다. 불필요해진 구성요소를 유지하는 것은 비용 낭비이자 복잡도 부채다.

### 14.1.2 다음에 불필요해질 구성요소는?

현재 구성요소를 "모델 의존 해소 가능성"으로 정렬한다:

| 구성요소 | 보완하는 한계 | 해소 시나리오 | 가능성 |
|---------|-------------|-------------|--------|
| Context Reset | 컨텍스트 누적 시 성능 저하 | 컨텍스트 윈도우 대폭 확장 또는 누적 내성 향상 | 높음 |
| Anti-rationalization | 자기합리화 패턴 | 모델이 자기합리화를 자체 인식/거부 | 중간 |
| Design Skill | AI slop 패턴 | 모델의 기본 디자인 감각 향상 | 중간 |
| Plateau Detection | 자율 종료 판단 부재 | 모델이 자체 수렴 판단 | 낮음 |
| Planner | 자발적 범위 확장 부재 | 모델이 지시 없이 야심적 범위 설정 | 낮음 |
| Clean-room 격리 | 자기평가 편향 (구조적) | 동일 컨텍스트에서 생성/평가 분리 가능 | 매우 낮음 |

**Context Reset이 가장 높은 이유**: 컨텍스트 윈도우 확장은 모델 회사들이 적극적으로 추구하는 방향이다. 윈도우가 충분히 확장되면, 3라운드 분량의 이력을 누적해도 성능 저하가 발생하지 않을 수 있다. 이 경우 context reset 대신 compaction으로 전환하는 것이 효율적일 수 있다.

**Clean-room 격리가 가장 낮은 이유**: 자기평가 편향은 모델 능력이 아닌 아키텍처의 구조적 속성이다. 아무리 강력한 모델이라도, 동일 컨텍스트에서 생성한 결과를 자체 평가하면 편향이 발생한다. 이 문제는 모델 능력 향상으로 해소되지 않으며, 아키텍처 수준의 격리만이 해법이다.

## 14.2 Feature Flag 기반 적응 전략

### 14.2.1 현재 구현의 feature flag

```typescript
interface FeatureFlags {
  planner: boolean
  designEvaluation: boolean
  aiWeaving: boolean
  contractGeneration: boolean
}
```

각 flag를 false로 설정하면 해당 구성요소가 비활성화된다. 코드 삭제 없이 동작을 변경할 수 있다.

### 14.2.2 향후 확장할 feature flag

현재 구현에 포함되지 않았으나, 향후 추가가 예상되는 flag:

```typescript
interface ExtendedFeatureFlags extends FeatureFlags {
  cleanRoomEvaluator: boolean    // false: Evaluator에 Generator 컨텍스트 전달
  antiRationalization: boolean   // false: anti-rationalization 프로토콜 제거
  contextReset: boolean          // false: 라운드 간 컨텍스트 유지
  plateauDetection: boolean      // false: maxRounds까지 항상 실행
  playwrightEvaluation: boolean  // false: Playwright 없이 정적 분석만
}
```

각 flag의 비활성화가 의미하는 것:

- `cleanRoomEvaluator: false` → Evaluator에게 Generator 컨텍스트를 전달. 자기평가 편향의 발현 여부를 관찰.
- `antiRationalization: false` → 자기합리화 패턴 목록을 제거. 모델이 자체적으로 합리화를 방지하는지 관찰.
- `contextReset: false` → 라운드 간 컨텍스트를 유지(compaction 적용). 성능 저하 여부 관찰.

### 14.2.3 환경변수 기반 설정

```bash
HARNESS_FEATURE_PLANNER=false npx harness-eng "Build a to-do app"
```

환경변수로 flag를 오버라이드할 수 있어, ablation study 실행 시 설정 파일 수정 없이 실험 조건을 변경할 수 있다.

## 14.3 하네스 조합 공간의 탐색

### 14.3.1 새 모델 출시 시 재검증 체크리스트

모델 업그레이드(예: Opus 4.6 → 5.0) 시 다음 절차를 수행한다:

**Phase 1: 기준선 확인 (1일)**
- [ ] 기존 벤치마크 앱 3종을 새 모델로 Solo 실행
- [ ] Solo 품질을 이전 모델의 Solo 품질과 비교
- [ ] Solo 능력 경계의 이동 방향 확인

**Phase 2: 하네스 기준선 (1일)**
- [ ] 동일 벤치마크 앱을 새 모델로 하네스 실행
- [ ] 하네스 품질을 이전 모델의 하네스 품질과 비교
- [ ] 비용 변화 확인

**Phase 3: Ablation study (3일)**
- [ ] 각 구성요소를 개별 비활성화하여 기여도 측정
- [ ] 기여도가 이전 모델 대비 감소한 구성요소 식별
- [ ] 기여도가 0에 근접한 구성요소를 제거 후보로 분류

**Phase 4: 새 구성요소 탐색 (2일)**
- [ ] 새 모델의 향상된 능력으로 가능해진 새 구성요소 식별
- [ ] 프로토타입 구현
- [ ] 기여도 측정

**Phase 5: 최적 조합 확정 (1일)**
- [ ] 제거 후보와 추가 후보를 반영한 새 조합 결정
- [ ] feature flag 업데이트
- [ ] 벤치마크로 최종 확인

### 14.3.2 구성요소 추가/제거 의사결정 프레임워크

구성요소의 추가 또는 제거를 결정하는 의사결정 트리:

```
구성요소 X에 대해:

1. Ablation에서 품질 변화가 있는가?
   ├── 없음 (±0.2점 이내) → 제거 후보
   └── 있음 → 2단계로

2. 품질 변화의 방향은?
   ├── 감소 (-0.3점 이상) → 유지
   └── 증가 → 구성요소가 품질을 저하시키고 있다. 제거 대상.

3. 비용 변화는?
   ├── 비용 절감 > 품질 감소 가치 → 제거 고려
   └── 비용 절감 < 품질 감소 가치 → 유지
```

"품질 감소 가치"의 정량화는 프로젝트 맥락에 의존한다. $10의 비용 절감을 위해 0.5점의 품질 감소를 감수할 것인가? 이 트레이드오프는 하네스 운영자가 결정한다.

## 14.4 장기적 전망

### 14.4.1 모델 능력 곡선과 하네스 가치 곡선

모델 능력이 선형적으로 향상된다고 가정하면:

```
가치
  │
  │  ╲ 하네스 복잡도
  │   ╲
  │    ╲          ╱ 하네스 달성 품질
  │     ╲       ╱
  │      ╲    ╱
  │       ╲ ╱
  │        ╳
  │       ╱ ╲
  │     ╱    ╲
  │   ╱
  │  ╱ Solo 달성 품질
  │
  └──────────────────── 모델 능력
```

모델 능력이 향상되면:
- Solo 달성 품질이 상승한다.
- 하네스 달성 품질도 상승한다(더 좋은 모델로 하네스를 실행하므로).
- 하네스의 복잡도(필요한 구성요소 수)는 감소한다.
- 두 곡선의 간격(하네스의 부가 가치)은 감소하되 0에 수렴하지 않는다.

### 14.4.2 왜 0에 수렴하지 않는가

Chapter 5의 원칙 4("조합 공간은 축소되지 않고 이동한다")가 이 질문에 대한 답이다.

모델이 현재 수준의 복잡한 앱을 Solo로 생성할 수 있게 되면, 하네스의 목표를 더 복잡한 앱으로 상향할 수 있다. 하네스의 가치는 "현재 모델이 Solo로 달성할 수 없는 수준의 복잡한 작업"에서 항상 존재한다. 그 수준이 시간에 따라 이동할 뿐이다.

또한 자기평가 편향은 모델 능력과 무관한 구조적 문제이므로, clean-room Evaluator의 가치는 모델 능력 향상에 의해 소멸하지 않는다.

### 14.4.3 하네스 엔지니어의 지속적 역할

모델이 진화해도 하네스 엔지니어의 역할은 사라지지 않는다. 역할의 내용이 변한다.

**현재**: 모델의 한계를 보완하는 구성요소를 설계하고 조합한다.
**미래**: 모델의 새로운 능력을 활용하는 구성요소를 설계하고, 기존 구성요소를 정리한다.

핵심 활동은 동일하다: ablation study로 각 구성요소의 가치를 측정하고, 최적 조합을 유지하며, 모델 진화에 따라 조합을 갱신한다. 도구가 변해도 방법론은 지속된다.
