# Chapter 5: V1→V2 진화에서 배우는 설계 원칙

## 5.1 진화의 배경

PDF가 기술하는 하네스는 단일 버전의 완성품이 아니다. V1(Opus 4.5 기반)에서 V2(Opus 4.6 기반)로의 진화 과정을 거쳤으며, 이 진화 과정 자체가 하네스 설계의 핵심 교훈을 담고 있다.

V1과 V2의 주요 차이:

| 항목 | V1 (Opus 4.5) | V2 (Opus 4.6) |
|------|--------------|--------------|
| 실행 시간 | 6시간 | 3시간 50분 |
| 비용 | $200 | $124.70 |
| 스프린트 구조 | 있음 | 제거됨 |
| 구성요소 수 | 더 많음 | 단순화됨 |

비용이 $200에서 $124.70으로 37.7% 감소했고, 실행 시간이 6시간에서 3시간 50분으로 36.1% 감소했다. 이 개선은 단순히 더 강력한 모델을 사용한 결과가 아니다. 모델 교체와 함께 아키텍처 단순화를 수행한 결과다.

이 진화 과정에서 PDF 저자가 도출한 4가지 설계 원칙을 분석한다.

## 5.2 원칙 1: 모델 능력 가정의 정기 재검증

PDF 원문:

> "Every component in a harness encodes an assumption about what the model can't do on its own, and those assumptions are worth stress testing."

### 5.2.1 원칙의 의미

하네스의 모든 구성요소는 암묵적 가정을 내포한다. "모델이 이 작업을 단독으로 수행할 수 없으므로, 이 구성요소가 보완한다"는 가정이다. 모델 능력이 향상되면 이 가정이 무효화될 수 있다.

스프린트 구조의 사례가 이 원칙을 가장 명확히 보여준다. V1에서 스프린트 구조는 Generator의 작업을 소규모 단위로 분해하여, 모델이 장기 일관성을 유지하지 못하는 한계를 보완하는 구성요소였다.

PDF 저자의 설명:

> "I started by removing the sprint construct entirely. The sprint structure had helped to decompose work into chunks for the model to work coherently. Given the improvements in Opus 4.6, there was good reason to believe that the model could natively handle the job without this sort of decomposition."

Opus 4.6의 장기 일관성 향상으로, 스프린트 분해 없이도 모델이 대규모 작업을 일관되게 수행할 수 있게 되었다. 스프린트 구조가 보완하던 모델 한계가 사실상 해소된 것이다.

### 5.2.2 적용 방법

각 구성요소에 대해 다음 질문을 주기적으로 적용한다:

1. **이 구성요소가 보완하는 모델 한계는 무엇인가?** 명시적으로 문서화한다.
2. **최신 모델에서 이 한계가 여전히 존재하는가?** 모델 업그레이드 시마다 검증한다.
3. **이 구성요소를 제거하면 산출물 품질이 어떻게 변화하는가?** 제거 실험(ablation)으로 확인한다.

이 질문을 구성요소별로 적용한 결과:

| 구성요소 | 보완하는 모델 한계 | 해소 가능성 |
|---------|------------------|-----------|
| Planner | 자발적 범위 확장 부재 | 낮음 (행동 패턴 문제) |
| Evaluator (clean-room) | 자기평가 편향 | 낮음 (구조적 문제) |
| Anti-rationalization | 자기합리화 패턴 | 중간 (프롬프트 의존) |
| Handoff 스키마 | 비정형 출력 불안정성 | 중간 |
| Context Reset | 컨텍스트 누적 성능 저하 | 높음 (윈도우 확장 시) |
| Plateau Detection | 자율 종료 판단 부재 | 낮음 |
| 디자인 스킬 | AI slop 패턴 | 중간 |

### 5.2.3 Feature Flag 기반 설계

이 원칙의 실무적 적용은 feature flag 내장이다. 모든 구성요소를 환경변수 또는 설정 파일로 개별 비활성화할 수 있도록 설계한다.

```
features:
  planner: true
  designEvaluation: true
  aiWeaving: true
  contractGeneration: true
```

이 설계가 보장하는 것은 "구성요소 제거의 저비용"이다. 구성요소가 불필요해졌다고 판단될 때, 코드 삭제 없이 설정 변경만으로 비활성화할 수 있다. 이는 ablation study(원칙 3)를 수행할 때도 필수적인 기반이다.

## 5.3 원칙 2: Evaluator 가치의 경계 비례

PDF 원문:

> "The evaluator is not a fixed yes-or-no decision. It is worth the cost when the task sits beyond what the current model does reliably solo."

### 5.3.1 원칙의 의미

Evaluator의 가치는 작업의 복잡도와 비례한다. 단순한 작업(예: "Hello World 앱")에서는 단일 에이전트의 산출물이 충분한 품질에 도달하므로, Evaluator의 추가 비용이 정당화되지 않는다. 복잡한 작업(예: "16기능 DAW 앱")에서는 단일 에이전트의 산출물 품질이 불충분하므로, Evaluator의 비용이 정당화된다.

이 관계를 도식화하면:

```
Evaluator 가치
    │
    │                    ╱
    │                  ╱
    │                ╱
    │              ╱
    │            ╱
    │          ╱
    │        ╱
    │      ╱
    │    ╱
    │──╱──────────────── 작업 복잡도
    │╱                   
    │
    └─────── Solo 능력 경계
```

"Solo 능력 경계"는 단일 에이전트가 충분한 품질의 산출물을 생성할 수 있는 복잡도의 상한선이다. 이 경계 이하에서는 Evaluator가 불필요하고, 이 경계 이상에서는 Evaluator가 필수적이다.

### 5.3.2 경계의 이동

중요한 점은 이 경계가 고정이 아니라는 것이다. 모델 능력이 향상되면 Solo 능력 경계가 우측으로 이동한다. Opus 4.5에서 경계가 "기능 5개" 수준이었다면, Opus 4.6에서는 "기능 8개" 수준으로 이동했을 수 있다.

이 이동은 Evaluator의 가치가 소멸한다는 의미가 아니다. 경계가 이동해도 경계 너머의 작업은 항상 존재한다. "16기능 DAW"가 Solo로 가능해진다면, 하네스의 목표를 "30기능 DAW"로 상향할 수 있다. Evaluator의 가치는 사라지지 않고 이동한다.

### 5.3.3 적응적 Evaluator 활성화

이 원칙의 실무적 적용은 작업 복잡도에 따른 적응적 Evaluator 활성화다. 향후 구현에서 고려할 수 있는 전략:

- 스펙의 기능 수가 임계값(예: 8개) 이하이면 Evaluator를 생략하고, 이상이면 활성화한다.
- 첫 라운드에서 Generator의 산출물을 간이 검증하고, 품질이 충분하면 추가 라운드를 생략한다.
- 비용 대비 품질 개선이 임계값 이하이면 Evaluator를 비활성화한다.

현재 구현에서는 이 적응적 전략을 적용하지 않는다. Evaluator는 항상 활성화된다. 적응적 활성화는 충분한 벤치마크 데이터가 축적된 후에 도입하는 것이 안전하다.

## 5.4 원칙 3: 체계적 단순화 (Ablation Study)

PDF 원문:

> "I moved to a more methodical approach, removing one component at a time and reviewing what impact it had on the final result."

### 5.4.1 급진적 단순화의 실패

PDF 저자는 V1에서 V2로의 전환 초기에 급진적 단순화를 시도했다. 여러 구성요소를 동시에 제거하고 결과를 관찰하는 접근이다. 이 접근은 실패했다.

실패의 원인은 효과의 중첩(confounding)이다. 구성요소 A와 B를 동시에 제거했을 때 품질이 저하되면, A의 효과인지 B의 효과인지 A와 B의 상호작용 효과인지 판별할 수 없다. 어떤 구성요소를 복원해야 하는지 결정할 근거가 없다.

### 5.4.2 점진적 단순화 (Ablation Study)

급진적 단순화의 실패 후, PDF 저자는 ablation study 방식으로 전환했다. 한 번에 하나의 구성요소만 제거하고, 그 제거의 효과를 측정한다. 각 실험에서 변수는 하나뿐이므로, 효과의 귀인이 명확하다.

Ablation study의 절차:

1. 기준선(baseline) 실행: 모든 구성요소가 활성화된 상태에서 벤치마크 앱을 생성하고 품질을 측정한다.
2. 구성요소 X를 비활성화하고 동일 벤치마크를 실행한다.
3. 품질 변화를 측정한다.
4. 구성요소 X를 복원하고, 구성요소 Y를 비활성화하여 반복한다.
5. 각 구성요소의 기여도를 비교하여 제거 후보를 선정한다.

### 5.4.3 적용 시 주의점

Ablation study에는 실무적 제약이 있다.

**비용 제약.** 각 실험이 하네스 1회 실행($124.70)을 소모한다. 7개 구성요소의 ablation study는 최소 8회 실행(기준선 1 + 제거 실험 7), 비용 $997.60이 필요하다. 이 비용은 단일 벤치마크 앱 기준이며, 통계적 유의성을 위해 앱 수를 늘리면 비용이 비례하여 증가한다.

**비결정성.** 에이전트의 출력은 비결정적이다. 동일 입력에 대해 다른 출력이 생성될 수 있다. 따라서 단일 실행의 품질 변화가 구성요소 제거의 효과인지 무작위 변동인지 판별하기 어렵다. 이상적으로는 각 조건에서 복수 실행하여 평균을 비교해야 하나, 비용 제약이 이를 제한한다.

**상호작용 효과.** 개별 ablation으로 검출할 수 없는 상호작용 효과가 존재할 수 있다. 구성요소 A 단독 제거와 B 단독 제거에서 품질 저하가 없더라도, A와 B를 동시에 제거하면 품질이 저하될 수 있다 (A와 B가 상호 보완적인 경우). 모든 조합을 테스트하면 이 문제가 해결되나, 조합 수가 2^n으로 폭발한다.

이러한 제약에도 불구하고, ablation study는 구성요소 가치를 체계적으로 평가하는 유일한 방법이다. 직관이나 추론에 기반한 제거보다 실험에 기반한 제거가 항상 신뢰성이 높다.

## 5.5 원칙 4: 하네스 조합 공간의 이동

PDF 원문:

> "The space of interesting harness combinations doesn't shrink as models improve. Instead, it moves."

### 5.5.1 원칙의 의미

이 원칙은 하네스 엔지니어링의 지속적 유효성에 대한 명제다. 모델 능력이 향상되면 일부 구성요소가 불필요해지지만, 동시에 새로운 구성요소가 가능해진다.

스프린트 구조의 사례를 확장하면: Opus 4.5에서는 장기 일관성 부족을 보완하기 위해 스프린트 분해가 필요했다. Opus 4.6에서는 장기 일관성이 향상되어 스프린트 분해가 불필요해졌다. 그러나 Opus 4.6의 향상된 능력으로 인해, 이전에는 불가능했던 새로운 구성요소(예: 더 복잡한 AI 기능 직조, 더 정밀한 디자인 평가)가 가능해졌다.

### 5.5.2 조합 공간의 도식화

```
모델 능력 축
    │
    │  ┌──────────────────────────┐
    │  │ Opus 4.6                 │
    │  │ - 스프린트 불필요         │
    │  │ + 정밀 디자인 평가 가능   │
    │  │ + 복잡 AI 직조 가능      │
    │  └──────────────────────────┘
    │
    │  ┌──────────────────────────┐
    │  │ Opus 4.5                 │
    │  │ + 스프린트 필요           │
    │  │ - 단순 디자인 평가만      │
    │  │ - 기본 AI 직조만          │
    │  └──────────────────────────┘
    │
    └─────────────────────────────── 시간
```

각 모델 세대에서 유효한 구성요소 집합이 다르다. V1의 최적 조합이 V2에서는 최적이 아니며, V2의 최적 조합이 V3에서는 다시 변경될 것이다.

### 5.5.3 설계적 함의

이 원칙의 설계적 함의는 "고정된 하네스를 목표로 하지 말라"는 것이다. 하네스는 특정 모델 세대에 최적화된 구성요소 조합이며, 모델이 진화하면 하네스도 함께 진화해야 한다.

이를 위한 설계 전략:

1. **조합 가능한 구성요소 집합 유지.** 각 구성요소가 독립적으로 활성화/비활성화 가능하도록 설계한다 (원칙 1의 feature flag).
2. **구성요소 간 결합도 최소화.** 구성요소 A의 존재가 구성요소 B의 동작에 영향을 미치지 않도록 설계한다. 이는 구성요소 제거 시 부작용을 방지한다.
3. **새 모델 출시 시 재검증 체크리스트.** 모델 업그레이드 시마다 각 구성요소의 유효성을 체계적으로 재검증하는 절차를 수립한다.

재검증 체크리스트의 예시:

- [ ] 각 구성요소의 "보완하는 모델 한계"를 검토한다.
- [ ] 해당 한계가 새 모델에서 해소되었는지 확인한다.
- [ ] 해소된 구성요소에 대해 ablation study를 실행한다.
- [ ] 새 모델의 향상된 능력으로 가능해진 새 구성요소를 식별한다.
- [ ] 새 구성요소를 프로토타입하고 기여도를 측정한다.
- [ ] 최적 조합을 갱신한다.

## 5.6 네 원칙의 통합

네 원칙은 독립적이지 않다. 상호 연결된 체계를 형성한다.

원칙 1(가정 재검증)이 "언제" 검토할 것인가를 결정한다. 모델 업그레이드가 트리거다.

원칙 2(경계 비례)가 "무엇을" 검토할 것인가를 결정한다. Solo 능력 경계의 이동을 추적한다.

원칙 3(체계적 단순화)이 "어떻게" 검토할 것인가를 결정한다. Ablation study로 각 구성요소의 기여도를 측정한다.

원칙 4(조합 공간 이동)가 검토의 "범위"를 결정한다. 제거만이 아니라 추가도 고려한다.
