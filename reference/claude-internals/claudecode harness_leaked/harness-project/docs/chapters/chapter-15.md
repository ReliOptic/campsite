# Chapter 15: 너머의 하네스

## 15.1 풀스택 앱 생성 너머의 적용 분야

이 책 전반에서 분석한 3-에이전트 하네스(Planner → Generator → Evaluator)는 풀스택 앱 자율 생성이라는 특정 작업에 최적화되었다. 그러나 하네스의 핵심 원리, 즉 범위 확장, 독립 평가, 자율 반복은 다른 작업 유형에도 적용 가능하다.

### 15.1.1 문서 생성 하네스

| 구성요소 | 풀스택 앱 하네스 | 문서 생성 하네스 |
|---------|----------------|----------------|
| Planner | 1문장 → 16기능 스펙 | 주제 → 챕터 구조 + 대상 독자 정의 |
| Generator | 스펙 → 실행 가능한 앱 | 구조 → 완성된 문서 |
| Evaluator | 디자인/기능 4축 평가 | 정확성/가독성/완성도 평가 |
| Clean-room | Generator 컨텍스트 차단 | 저자 의도 차단, 독자 관점으로만 평가 |
| Handoff | must_fix/preserve + REFINE/PIVOT | 수정 항목/보존 항목 + 구조 유지/재구성 |

문서 생성에서 clean-room 격리의 가치: 문서 작성자(Generator)는 자신이 작성한 문서를 "이해하기 쉽다"고 판단하는 경향이 있다. 내용을 이미 알고 있기 때문이다. Evaluator가 저자의 의도 없이 문서만 읽으면, 실제 독자가 경험하는 가독성을 더 정확히 측정할 수 있다.

문서 생성 하네스에서 PIVOT의 의미: 풀스택 앱 하네스에서 PIVOT은 미학적 방향의 전면 재설계였다. 문서 생성에서 PIVOT은 문서 구조의 전면 재편성이다. "튜토리얼 형식이 적합하지 않으니 레퍼런스 형식으로 전환하라"와 같은 지시가 된다.

### 15.1.2 코드 리뷰 하네스

코드 리뷰 하네스는 3-에이전트 구조를 코드 품질 보증에 적용한다.

| 구성요소 | 역할 |
|---------|------|
| Planner | diff를 분석하여 리뷰 관점(보안, 성능, 가독성, 테스트) 결정 |
| Generator | 각 관점에서 구체적 리뷰 코멘트 생성 |
| Evaluator | 리뷰 코멘트의 품질 평가 (오탐 제거, 심각도 보정) |

코드 리뷰에서 자기평가 편향의 발현: Generator가 "이 코드는 보안 취약점이 없다"고 판단한 후, Evaluator 역할에서 "Generator의 보안 분석이 적절하다"고 동조하는 패턴. Clean-room Evaluator는 Generator의 분석 없이 코드 자체만으로 보안 취약점을 독립 탐색한다.

이 하네스에서 반복 루프의 의미: 1차 리뷰(Generator)가 표면적 이슈만 발견하면, Evaluator가 "심층 분석 부족"을 지적하고, 2차 리뷰에서 Generator가 더 깊은 분석을 수행한다. 리뷰의 깊이가 라운드마다 증가하는 구조다.

### 15.1.3 데이터 파이프라인 하네스

데이터 파이프라인 구축에 하네스를 적용한다.

| 구성요소 | 역할 |
|---------|------|
| Planner | 데이터 소스, 변환 요구사항, 출력 형식을 스펙으로 확장 |
| Generator | ETL 파이프라인 코드 생성, 스키마 정의, 테스트 데이터 생성 |
| Evaluator | 데이터 정합성 검증, 성능 벤치마크, 에지 케이스 테스트 |

데이터 파이프라인에서 Evaluator의 특수성: 풀스택 앱의 Evaluator가 시각적 품질을 주관적으로 평가하는 것과 달리, 데이터 파이프라인의 Evaluator는 데이터 정합성을 객관적으로 검증할 수 있다. 입력 데이터와 출력 데이터의 관계가 수학적으로 정의 가능하기 때문이다. 이는 Evaluator의 신뢰성이 더 높아짐을 의미한다.

## 15.2 멀티 하네스 조합

### 15.2.1 개념

단일 하네스가 하나의 작업 유형에 최적화되어 있다면, 복수 하네스를 조합하여 복합 작업을 수행할 수 있다.

예시: "풀스택 앱 + 문서"를 생성하는 복합 하네스.

```
사용자 프롬프트
    │
    ├──► [풀스택 앱 하네스] ──► 완성된 앱
    │
    └──► [문서 생성 하네스] ──► 사용자 가이드, API 문서
              ▲
              │ 앱 코드 참조
```

두 하네스가 병렬 실행되되, 문서 생성 하네스가 앱 하네스의 산출물(코드)을 참조하는 구조다. 완전 병렬이 아닌 부분 의존(partial dependency) 관계다.

### 15.2.2 조합 시 주의점

**비용 폭발.** 각 하네스가 $124.70이면 2개 조합은 $250 이상이다. 하네스 수가 증가하면 비용이 선형으로 증가한다.

**조합 간 일관성.** 앱 하네스의 Planner가 "어두운 테마"를 결정했는데, 문서 하네스의 Generator가 "밝은 테마"의 스크린샷을 생성하면 일관성이 파괴된다. 조합 간 공유 스펙이 필요하다.

**종료 조건의 동기화.** 앱 하네스가 3라운드 만에 PASS를 달성했는데 문서 하네스가 아직 2라운드 진행 중이면, 전체 작업의 완료 시점 결정이 복잡해진다.

## 15.3 하네스 엔지니어의 역할 정의

### 15.3.1 프롬프트 엔지니어와의 차이

| 영역 | 프롬프트 엔지니어 | 하네스 엔지니어 |
|------|-----------------|---------------|
| 대상 | 단일 모델 호출 | 복수 에이전트 시스템 |
| 최적화 대상 | 프롬프트 텍스트 | 에이전트 조합 + 프롬프트 + 평가 |
| 평가 방법 | 출력 품질 검토 | ablation study + 벤치마크 |
| 비용 관심 | 토큰 수 최소화 | 전체 시스템 비용 최적화 |
| 주요 도구 | 프롬프트, few-shot 예시 | 스키마, 격리 설계, 비용 추적기 |

프롬프트 엔지니어링은 하네스 엔지니어링의 부분 집합이다. 하네스 엔지니어는 각 에이전트의 프롬프트를 설계하지만, 그것만으로는 부족하다. 에이전트 간 데이터 흐름, 격리 경계, 비용 구조, 수렴 메커니즘을 함께 설계해야 한다.

### 15.3.2 3대 역량 영역

하네스 엔지니어는 세 영역의 교차점에 위치한다.

**1. 시스템 설계**
- 에이전트 간 경계와 책임 분배
- 데이터 흐름과 스키마 설계
- 격리 모델 (clean-room, context reset)
- 비용 구조와 예산 제어

이 영역은 소프트웨어 아키텍트의 역량과 유사하다. 분산 시스템의 서비스 경계 설계, API 계약 설계와 동일한 사고 방식이 적용된다.

**2. 프롬프트 설계**
- 각 에이전트의 시스템 프롬프트
- Anti-rationalization 지시
- Few-shot 캘리브레이션
- Feature-specific 지시 (anti-slop, AI weaving)

이 영역은 프롬프트 엔지니어의 역량이다. 차이점은 프롬프트가 격리된 환경에서 동작하므로, 단일 프롬프트의 최적화가 아닌 프롬프트 간 상호작용을 고려해야 한다.

**3. 평가 설계**
- 평가 기준 정의 (4축 + 임계값)
- Evaluator 프롬프트의 교정
- Ablation study 설계와 실행
- 벤치마크 선정과 성능 측정

이 영역은 ML 엔지니어의 평가(evaluation) 역량과 유사하다. 메트릭 선정, 실험 설계, 통계적 유의성 판단이 필요하다.

### 15.3.3 실무적 활동

하네스 엔지니어의 일상적 활동:

**구축 단계:**
- 작업 유형에 적합한 에이전트 조합 설계
- 각 에이전트의 프롬프트 작성과 스키마 정의
- 비용 제어 메커니즘 구현
- 테스트 전략 수립

**운영 단계:**
- 벤치마크 실행과 성능 추적
- 프롬프트 미세 조정 (산출물 품질 기반)
- 비용 최적화 (불필요한 라운드 감소, 프롬프트 압축)
- 에러 패턴 분석과 복구 전략 개선

**적응 단계:**
- 새 모델 출시 시 재검증 체크리스트 수행
- Ablation study로 구성요소 가치 재평가
- 불필요한 구성요소 제거, 새 구성요소 추가
- 최적 조합 갱신

## 15.4 결론

이 책은 하네스 엔지니어링을 이론(Part I), 실전(Part II), 응용(Part III)의 세 관점에서 다루었다.

Part I에서 확립한 이론: 에이전트 런타임과 하네스 응용의 계층 분리, 3-에이전트 아키텍처의 데이터 흐름, 자기평가 편향과 clean-room 격리, context anxiety와 structured handoff, V1→V2 진화의 4대 설계 원칙.

Part II에서 구현한 실전: Agent SDK 기반 프로젝트 구조, Zod 스키마 4종, Planner/Generator/Evaluator의 프롬프트와 코드, Orchestrator의 GAN 루프와 CostTracker, 3계층 테스트 전략.

Part III에서 탐색한 응용: 벤치마크와 ablation study, 모델 진화 대응, 풀스택 앱 너머의 적용 분야, 하네스 엔지니어의 역할.

하네스 엔지니어링의 핵심 명제를 다시 정리한다:

1. **하네스의 구성요소는 모델 능력의 가정을 인코딩한다.** 가정이 무효화되면 구성요소를 제거한다.
2. **자기평가 편향은 구조적 문제다.** 모델 능력 향상으로 해소되지 않으며, 아키텍처 수준의 격리가 필요하다.
3. **하네스 조합 공간은 축소되지 않고 이동한다.** 모델이 진화해도 하네스의 가치는 사라지지 않는다.
4. **체계적 단순화만이 유효하다.** 직관이 아닌 ablation study에 기반하여 구성요소를 관리한다.

이 명제들은 특정 모델 세대에 종속되지 않는다. Claude Opus 4.6이 10.0이 되어도, GPT가 6.0이 되어도, 이 원칙은 유효하다. 원칙이 적용되는 구체적 구성요소는 변하겠지만, 원칙 자체는 에이전트 시스템의 구조적 속성에 기반하므로 지속된다.

하네스 엔지니어링은 아직 초기 단계의 분야다. 이 책이 다룬 3-에이전트 구조는 가능한 많은 조합 중 하나에 불과하다. 5-에이전트 하네스, 계층적 하네스, 적응적 하네스 등 탐색되지 않은 조합 공간이 방대하게 존재한다. 이 공간을 체계적으로 탐색하는 것이 하네스 엔지니어의 과제이자 기회다.
