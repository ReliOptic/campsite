# Chapter 1: 하네스란 무엇인가

## 1.1 에이전트 런타임과 하네스 응용의 계층 분리

AI 에이전트 시스템을 논의할 때 가장 먼저 명확히 해야 할 구분이 있다. **에이전트 런타임**(agent runtime)과 **하네스 응용**(harness application)은 동일 계층이 아니다.

에이전트 런타임은 범용 실행 환경이다. 사용자의 지시를 수신하고, 도구를 호출하며, 결과를 반환하는 기반 인프라다. Claude Code CLI가 이 범주에 속한다. 모든 사용자, 모든 작업 유형을 대상으로 설계되었으며, 안정성이 야심보다 우선한다.

하네스 응용은 이 런타임 위에 구축된 특수 목적 시스템이다. 특정 작업 유형(이 책에서는 풀스택 앱 자율 생성)에 최적화된 에이전트 조합을 정의하고, 그 조합의 실행을 제어한다. 야심이 안정성보다 우선한다.

| 속성 | 에이전트 런타임 | 하네스 응용 |
|------|----------------|------------|
| 대상 사용자 | 모든 개발자 | 특정 작업 수행자 |
| 작업 범위 | 범용 | 단일 목적 |
| 설계 우선순위 | 안정성 > 야심 | 야심 > 안정성 |
| 비용 모델 | 사용자 주도 (예측 가능) | 시스템 주도 (가변적) |
| 에이전트 수 | 단일 | 복수 (3개 이상) |

이 계층 분리를 이해하지 못하면 하네스 설계에서 근본적인 오류를 범하게 된다. 범용 런타임에 하네스 논리를 내장하려는 시도는 과적합(overfitting)이며, 반대로 하네스를 범용 런타임으로 확장하려는 시도는 과소 특화(under-specialization)다. 두 계층은 Agent SDK라는 인터페이스를 통해 연결되지만, 설계 결정은 각 계층에서 독립적으로 내려진다.

Anthropic의 PDF 문서가 기술한 하네스는 Claude Agent SDK로 Claude Code의 프리미티브(파일 읽기, 파일 쓰기, 셸 실행, 웹 검색 등)를 조합하여 구축한 상위 응용이다. "런타임에 없는 기능"이 아니라 "런타임 위에 쌓은 응용"으로 이해해야 정확하다.

## 1.2 단일 에이전트의 한계

하네스가 왜 필요한가를 이해하려면, 먼저 단일 에이전트가 복잡한 작업에서 보이는 체계적 한계를 분석해야 한다.

### 1.2.1 과소 범위 설정 (Under-scoping)

단일 에이전트에게 "DAW 앱을 만들어라"라고 지시하면, 에이전트는 즉시 코드 작성을 시작한다. 스펙 작성 단계를 건너뛰고 직접 구현에 돌입하는 것이다. 결과적으로 에이전트가 해석한 범위는 사용자가 기대한 범위보다 현저히 좁다.

PDF 저자의 관찰을 인용한다:

> "Without the planner, the generator under-scoped: given the raw prompt, it would start building without first speccing its work."

이 현상은 모델의 능력 부족이 아니라 **작업 분해의 부재**에서 기인한다. 모델은 "1문장 프롬프트를 16개 기능으로 확장하라"는 명시적 지시가 있으면 충분히 야심적인 스펙을 생성할 수 있다. 문제는 그 지시가 없을 때 자발적으로 스펙 단계를 삽입하지 않는다는 점이다.

과소 범위 설정의 결과물은 기술적으로 동작하지만 제품으로서 불완전하다. "DAW 앱"이 오디오 트랙 1개와 재생 버튼만 있는 프로토타입으로 끝나는 것이 전형적 사례다.

### 1.2.2 자기평가 편향 (Self-Evaluation Bias)

단일 에이전트에게 자신의 작업을 평가하게 하면 체계적으로 높은 점수를 부여한다. PDF 저자는 이 현상을 직접 관찰했다:

> "Out of the box, Claude is a poor QA agent. In early runs, I watched it identify legitimate issues, then talk itself into deciding they weren't a big deal and approve the work anyway."

이 편향은 4가지 자기합리화 패턴으로 분류된다:

1. **코드 독해 대체**: "코드를 읽어본 바로는 정확해 보인다." 실제 실행 없이 코드 읽기로 검증을 대체한다.
2. **심각도 하향**: "사소한 문제이므로 통과시키겠다." 실제 기준 미달인 결함을 임의로 경미하다고 재분류한다.
3. **총괄적 승인**: "전반적으로 좋은 품질이다." 개별 기준 미달을 총점으로 상쇄하여 합격시킨다.
4. **구현자 신뢰**: "구현자의 테스트가 이미 통과한다." 자신이 작성한 테스트의 통과를 독립적 검증으로 간주한다.

이 편향은 모델의 의도적 기만이 아니다. 동일 컨텍스트 내에서 생성과 평가를 수행할 때 발생하는 구조적 문제다. 생성 과정에서 축적된 의도, 설계 결정, 디버깅 이력이 평가 시점의 판단을 오염시킨다.

### 1.2.3 컨텍스트 불안 (Context Anxiety)

장시간 실행되는 작업에서 에이전트의 컨텍스트 윈도우가 한계에 근접하면 행동 패턴이 변화한다. 에이전트는 작업을 조기에 종료하거나, 기능을 의도적으로 축소하거나, 품질 기준을 자의적으로 완화한다. PDF에서는 이 현상을 "context anxiety"로 명명한다.

컨텍스트 불안의 핵심 메커니즘은 다음과 같다. 에이전트가 이전 라운드의 대화 이력, 디버깅 로그, 실패한 시도 등을 컨텍스트에 누적하면, 실제 작업에 사용할 수 있는 유효 컨텍스트가 축소된다. 에이전트는 이 축소를 감지하고 방어적 행동을 취한다. "충분히 좋다"고 자체 판단하여 추가 반복을 회피하는 것이 전형적 증상이다.

이 세 가지 한계, 과소 범위 설정, 자기평가 편향, 컨텍스트 불안은 모델의 능력 한계가 아니라 **단일 에이전트 아키텍처의 구조적 한계**다. 더 강력한 모델을 사용해도 이 문제들은 완화될 수 있으나 해소되지는 않는다. 해법은 아키텍처 수준에서 찾아야 한다.

## 1.3 하네스가 해결하는 3가지 문제

하네스는 단일 에이전트의 세 가지 구조적 한계 각각에 대응하는 전문 에이전트를 배치하여 문제를 해결한다.

### 1.3.1 범위 확장: Planner

Planner 에이전트는 사용자의 1-4문장 프롬프트를 구조화된 제품 스펙으로 확장한다. "DAW 앱을 만들어라"라는 프롬프트가 16개 기능, 디자인 언어, 기술 아키텍처, AI 통합 방안을 포함하는 완전한 스펙으로 변환된다.

Planner의 설계 원칙은 PDF에서 다음과 같이 기술된다:

> "I prompted it to be ambitious about scope... stay focused on product context and high level technical design."

주목할 점은 Planner가 기술적 상세를 회피하도록 설계되었다는 것이다. Planner가 세부 구현 방식까지 지정하면, 그 지정이 오류일 경우 Generator 전체에 오류가 전파(cascade)된다. Planner는 "무엇을"에 집중하고 "어떻게"는 Generator에 위임한다.

비용 측면에서 Planner는 전체 하네스 비용의 0.4%를 차지한다. PDF 기준 $0.46. 이 미미한 비용으로 과소 범위 문제를 구조적으로 해결한다는 점에서 비용 효율이 극히 높은 구성요소다.

### 1.3.2 품질 보증: Evaluator

Evaluator 에이전트는 Generator의 산출물을 독립적으로 평가한다. 핵심은 **clean-room 격리**다. Evaluator는 Generator의 컨텍스트에 접근하지 못한다. Evaluator가 수신하는 정보는 원래 스펙, 완료 기준 계약, 생성된 코드, 실행 중인 앱뿐이다. Generator가 왜 특정 결정을 내렸는지, 어떤 디버깅을 거쳤는지, 어떤 대안을 검토했는지에 대한 정보는 완전히 차단된다.

이 격리가 자기평가 편향을 해소하는 메커니즘이다. Evaluator는 Generator의 의도를 알지 못하므로, 의도에 동조하는 합리화가 구조적으로 불가능하다. 평가는 오직 스펙 대비 산출물의 객관적 비교로만 수행된다.

Evaluator의 비용은 전체의 8.3% (PDF 기준 $10.39/3라운드). 이 비용으로 독립적 품질 보증을 확보하는 것은 소프트웨어 공학의 코드 리뷰 원칙과 정확히 일치한다. 코드 작성자와 리뷰어가 분리되어야 리뷰가 유효한 것처럼, Generator와 Evaluator가 분리되어야 평가가 유효하다.

### 1.3.3 자율 반복: Orchestrator

Orchestrator는 Planner, Generator, Evaluator 간의 데이터 흐름을 제어하고, 반복 루프의 진입 및 종료 조건을 관리한다. GAN(Generative Adversarial Network)의 Generator-Discriminator 구조에서 영감을 받았으나, 적대적(adversarial)이 아닌 협력적(cooperative) 관계로 설계되었다.

Orchestrator가 관리하는 반복 루프는 다음과 같다:

1. Planner가 스펙을 생성한다.
2. Generator가 스펙 기반으로 앱을 빌드한다.
3. Evaluator가 산출물을 평가하고 verdict를 결정한다.
4. verdict가 PASS이면 루프를 종료한다.
5. verdict가 FAIL 또는 PARTIAL이면 Evaluator의 피드백을 structured handoff로 변환하여 Generator에 전달한다.
6. Generator가 handoff를 기반으로 수정 빌드를 수행한다.
7. 3-6을 반복한다.

루프 종료 조건은 세 가지다: PASS 달성, 점수 정체(plateau) 감지, 예산 소진. 이 세 조건이 무한 반복을 방지하는 안전장치다.

컨텍스트 불안 문제는 Orchestrator의 **context reset** 전략으로 해결된다. 각 라운드에서 Generator와 Evaluator는 새로운 agent() 호출로 실행되며, 이전 라운드의 컨텍스트를 계승하지 않는다. 라운드 간 정보 전달은 고정 스키마의 handoff 파일로만 수행된다. 이는 컨텍스트 요약(compaction)과 본질적으로 다른 접근이다. 요약은 정보를 축소하면서 손실을 감수하지만, reset은 전달할 정보를 명시적으로 선택한다.

## 1.4 사례 연구: DAW 앱, Solo vs 하네스

PDF가 제시하는 핵심 사례는 DAW(Digital Audio Workstation) 앱 생성이다. 동일한 프롬프트("Build a DAW application")를 단일 에이전트(Solo)와 3-에이전트 하네스로 실행한 결과를 비교한다.

### Solo 실행

| 항목 | 수치 |
|------|------|
| 실행 시간 | 20분 |
| 비용 | $9 |
| 에이전트 수 | 1 |
| 반복 횟수 | 0 (단일 패스) |

Solo 실행의 산출물은 기본적인 오디오 재생 기능을 갖춘 웹 앱이다. 기술적으로 동작하지만, 트랙 편집, 이펙트 체인, 미디 지원, 프로젝트 저장/로드 등 DAW의 핵심 기능이 대부분 누락되어 있다. 과소 범위 설정의 전형적 결과다.

### 하네스 실행 (V2)

| 항목 | 수치 |
|------|------|
| 실행 시간 | 3시간 50분 |
| 비용 | $124.70 |
| 에이전트 수 | 3 (Planner + Generator + Evaluator) |
| 반복 횟수 | 3라운드 |

하네스 실행의 산출물은 16개 기능을 갖춘 완전한 DAW 앱이다. Planner가 범위를 확장하고, Generator가 3라운드에 걸쳐 구현을 개선하며, Evaluator가 각 라운드마다 독립 평가를 수행했다.

### 비용 대비 가치 분석

비용은 $9 대 $124.70으로 13.9배 차이다. 그러나 산출물의 기능 수는 2-3개 대 16개로 5-8배 차이이며, 디자인 품질과 완성도는 정량 비교가 어려울 정도로 격차가 크다.

이 비교에서 핵심 질문은 "13.9배의 비용이 정당한가"가 아니다. 진정한 질문은 "$9로 생성한 산출물이 제품으로서 가치가 있는가"다. 만약 $9 산출물이 추가 수작업 없이는 배포 불가능하다면, 그 $9는 최종 산출물에 대한 기여가 0에 가깝다. 반면 $124.70 산출물이 최소한의 수정으로 배포 가능하다면, 그 비용 전체가 최종 산출물에 기여한다.

하네스의 비용 구조에서 주목할 점은 오케스트레이션 오버헤드가 미미하다는 것이다. Planner 비용은 0.4%, Evaluator 비용은 8.3%다. 전체 비용의 91.3%는 Generator, 즉 실제 코드 생성에 소비된다. 하네스 아키텍처가 추가하는 간접 비용은 전체의 8.7%에 불과하다. 이 8.7%로 범위 확장, 독립 품질 평가, 자율 반복이라는 세 가지 구조적 이점을 확보한다.

### 비용 수렴 현상

라운드별 비용을 관찰하면 수렴 패턴이 나타난다:

| 라운드 | Build 비용 | QA 비용 |
|--------|-----------|---------|
| Round 1 | $71.08 | $3.24 |
| Round 2 | $36.89 | $3.09 |
| Round 3 | $5.88 | $4.06 |

Generator의 비용이 라운드마다 급격히 감소한다. Round 1에서 $71.08이던 비용이 Round 3에서 $5.88로 떨어진다. 이는 각 라운드에서 수정 범위가 축소되기 때문이다. Round 1은 전체 앱을 처음부터 구축하지만, Round 3는 Evaluator가 지적한 특정 항목만 수정한다.

반면 Evaluator의 비용은 라운드에 관계없이 $3-4로 일정하다. 평가 범위가 항상 전체 앱이기 때문이다. 이 비대칭은 하네스 비용 제어의 핵심 통찰이다. 라운드 수를 늘려도 추가 비용의 대부분은 Generator에서 발생하며, 그 Generator 비용은 라운드마다 감소한다. 3라운드를 넘어가면 추가 비용 대비 품질 개선이 체감적으로 감소하는 수확 체감(diminishing returns) 구간에 진입한다.

## 1.5 하네스 엔지니어링이란

이 책의 제목이자 핵심 주장은 "하네스 엔지니어링"이 독립적인 기술 분야라는 것이다.

프롬프트 엔지니어링이 단일 모델 호출의 입력을 최적화하는 기술이라면, 하네스 엔지니어링은 **복수 에이전트의 조합, 데이터 흐름, 품질 제어, 비용 관리를 설계하는 기술**이다. 프롬프트 엔지니어링이 하네스 엔지니어링의 부분 집합인 것이지, 하네스 엔지니어링이 프롬프트 엔지니어링의 확장인 것이 아니다.

하네스 엔지니어는 다음 세 영역의 교차점에 위치한다:

1. **시스템 설계**: 에이전트 간 경계, 데이터 흐름, 격리 모델, 비용 구조
2. **프롬프트 설계**: 각 에이전트의 시스템 프롬프트, anti-rationalization 지시, few-shot 캘리브레이션
3. **평가 설계**: 평가 기준 정의, 임계값 설정, 수렴 판단, ablation study

이 책은 이 세 영역을 이론(Part I)과 실전(Part II)으로 나누어 다루며, Part III에서 하네스의 응용과 진화를 논의한다. Part II의 구현은 Claude Agent SDK를 사용하여 독립 프로젝트로 수행하며, 독자가 직접 실행하고 변형할 수 있는 완전한 코드를 제공한다.
