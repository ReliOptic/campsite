# Chapter 13: 벤치마크와 평가

## 13.1 하네스 성능 측정 방법

Chapter 12에서 테스트가 검증할 수 없는 영역을 분석했다. 벤치마크는 그 영역, 즉 "하네스가 실제로 유효한 결과를 생성하는가"를 측정하는 방법이다.

### 13.1.1 Solo vs 하네스 A/B 비교

가장 기본적인 벤치마크는 동일 프롬프트를 Solo(단일 에이전트)와 하네스(3-에이전트)로 실행하여 산출물을 비교하는 것이다.

비교 축:

| 측정 항목 | 측정 방법 |
|----------|----------|
| 기능 수 | 스펙 대비 구현된 기능 수 |
| 디자인 품질 | 인간 평가자의 1-10 채점 |
| 기능 완성도 | 각 기능의 정상 동작 여부 |
| 비용 | cost-report.json의 totalUsd |
| 실행 시간 | 전체 실행 시간 |

A/B 비교의 기본 프로토콜:

1. 벤치마크 앱 프롬프트를 선정한다 (예: "Build a DAW application").
2. Solo 실행: 동일 모델(Opus 4.6), 동일 도구, 단일 agent() 호출.
3. 하네스 실행: 동일 모델, 3-에이전트 루프, 기본 설정.
4. 두 산출물을 동일 기준으로 평가한다.

### 13.1.2 비용 대비 품질 곡선

단순 비교를 넘어, 비용과 품질의 관계를 곡선으로 분석한다.

```
품질
(점수)
10 │
   │                      ● 하네스 R3
 8 │                 ●
   │            ●    하네스 R2
 6 │       ●
   │  ● Solo   하네스 R1
 4 │
   │
 2 │
   │
   └──────────────────────────── 비용 ($)
   0    9   50   75  100  125
```

이 곡선에서 관찰해야 할 특성:

**초기 급상승 구간 (Solo → R1).** $9에서 $80으로 비용이 9배 증가하면서 품질이 급격히 상승한다. 이 구간의 비용 효율이 가장 높다. Planner의 범위 확장이 이 급상승의 주요 원인이다.

**수확 체감 구간 (R2 → R3).** $115에서 $125로 비용이 소폭 증가하면서 품질 개선이 완만해진다. 이 구간에서 추가 라운드의 한계 비용 대비 한계 품질 개선을 평가해야 한다.

**평탄 구간.** 3라운드 이후 추가 라운드에서 품질 개선이 거의 없는 구간. Plateau detection이 이 구간의 진입을 감지하여 불필요한 비용 소비를 방지한다.

## 13.2 Ablation Study 실행

### 13.2.1 목적

Ablation study는 각 구성요소의 기여도를 독립적으로 측정한다. 구성요소를 하나씩 제거하고 산출물 품질의 변화를 관찰한다.

### 13.2.2 실행 프로토콜

```
기준선: 모든 구성요소 활성화
  → 벤치마크 앱 3종 실행, 평균 품질 측정

실험 1: Planner 비활성화
  → 사용자 프롬프트를 Generator에 직접 전달
  → 동일 벤치마크 실행, 품질 측정

실험 2: Evaluator 비활성화
  → Generator 1회 실행 후 종료 (피드백 루프 없음)
  → 동일 벤치마크 실행, 품질 측정

실험 3: Anti-rationalization 제거
  → Evaluator 프롬프트에서 anti-rationalization 프로토콜 삭제
  → 동일 벤치마크 실행, 품질 측정

실험 4: Design Skill 제거
  → Planner/Generator 프롬프트에서 디자인 스킬 참조 삭제
  → 동일 벤치마크 실행, 품질 측정

실험 5: AI Weaving 제거
  → features.aiWeaving = false
  → 동일 벤치마크 실행, 품질 측정

실험 6: Clean-room 격리 해제
  → Evaluator에게 Generator 컨텍스트를 전달
  → 동일 벤치마크 실행, 품질 측정

실험 7: Plateau Detection 제거
  → 항상 maxRounds까지 실행
  → 동일 벤치마크 실행, 품질 및 비용 측정
```

### 13.2.3 결과 해석 프레임워크

각 실험의 결과를 다음 형식으로 기록한다:

| 구성요소 | 품질 변화 | 비용 변화 | 해석 |
|---------|----------|----------|------|
| Planner | -2.5점 | -$0.46 | 범위 축소로 인한 품질 저하. 비용 절감 미미. 유지 필수. |
| Evaluator | -1.8점 | -$10.39 | 피드백 부재로 R1 수준에서 정체. 유지 권장. |
| Anti-rational. | -0.7점 | $0 | Evaluator 통과율 상승, 실제 품질 저하. 유지 권장. |
| Design Skill | -1.2점 | $0 | AI slop 패턴 증가. 유지 권장. |
| AI Weaving | -0.3점 | -$2 | AI 기능 부재. 프로젝트에 따라 선택적. |
| Clean-room | -0.5점 | $0 | 자기평가 편향 부분 발현. 유지 권장. |
| Plateau Det. | +0.1점 | +$15 | 미세 품질 개선, 비용 증가. 유지 권장. |

(위 수치는 예시이며, 실제 ablation 결과에 따라 다르다.)

### 13.2.4 핵심 질문

각 구성요소에 대해:

> "이 구성요소가 보완하는 모델 한계는 무엇인가?"

이 질문의 답이 ablation 결과를 해석하는 열쇠다. 품질 저하가 발생하면 해당 구성요소가 보완하는 모델 한계가 여전히 존재함을 의미한다. 품질 변화가 없으면 해당 한계가 해소되었거나 구성요소의 보완이 무효함을 의미한다.

## 13.3 벤치마크 앱 선정 기준

### 13.3.1 복잡도 변화

벤치마크 앱은 복잡도가 다양해야 한다. 단일 복잡도의 앱만 사용하면, 하네스의 성능이 해당 복잡도에서만 측정된다.

| 복잡도 | 예시 | 기능 수 | 예상 Solo 품질 |
|--------|------|---------|--------------|
| 낮음 | To-do 앱 | 3-5 | 높음 |
| 중간 | 프로젝트 관리 앱 | 8-12 | 중간 |
| 높음 | DAW 앱 | 15-20 | 낮음 |

Chapter 5의 원칙 2(Evaluator 가치의 경계 비례)를 검증하는 데 이 다양성이 필수적이다. 복잡도가 낮은 앱에서는 하네스의 가치가 미미할 것으로 예상되며, 복잡도가 높은 앱에서는 하네스의 가치가 극대화될 것으로 예상된다.

### 13.3.2 AI 통합 유무

AI 기능이 포함된 앱과 포함되지 않은 앱을 모두 벤치마크에 포함한다. AI Weaving 구성요소의 기여도를 평가하기 위함이다.

### 13.3.3 반복 가능성

벤치마크 앱의 프롬프트는 고정하여 반복 실행할 수 있어야 한다. "Build a DAW application"과 같이 명확하고 재현 가능한 프롬프트를 사용한다. "이전에 만들었던 것과 비슷한 앱"처럼 컨텍스트 의존적인 프롬프트는 벤치마크에 부적합하다.

## 13.4 평가 메트릭

### 13.4.1 자동 메트릭

비용과 시간은 cost-report.json에서 자동으로 수집된다:
- 총 비용 (totalUsd)
- 라운드 수
- 라운드별 비용 감소율
- 예산 활용률 (budgetUtilization)
- 총 실행 시간

### 13.4.2 반자동 메트릭

스펙 충족도는 자동 수집과 수동 확인의 조합으로 측정한다:
- 스펙의 기능 수 대비 구현된 기능 수 (자동: 파일 존재 확인)
- 각 기능의 실제 동작 여부 (수동: 앱 실행 확인)
- 빌드 성공 여부 (자동: npm build 실행)

### 13.4.3 수동 메트릭

디자인 품질과 사용자 경험은 인간 평가자가 측정한다:
- 4축 채점 (design_quality, originality, craft, functionality)
- 전반적 인상 (제품으로서의 완성도)
- AI slop 존재 여부

인간 평가의 한계: 비용이 높고 주관적이다. 이 한계를 인식하되, 디자인 품질의 최종 판단은 인간에게만 가능하다는 현실을 수용한다. Evaluator 에이전트의 채점은 인간 평가의 대리(proxy)이지 대체(substitute)가 아니다.
