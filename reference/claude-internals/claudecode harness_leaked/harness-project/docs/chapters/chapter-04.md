# Chapter 4: Context Anxiety와 라운드 간 정보 전달

## 4.1 Context Anxiety 현상

장시간 실행되는 에이전트 작업에서 관찰되는 특이한 행동 패턴이 있다. 에이전트의 컨텍스트 윈도우가 한계에 근접하면, 에이전트는 작업의 완성도와 무관하게 종료를 시도한다. PDF에서는 이 현상을 "context anxiety"라고 명명한다.

이 현상의 발현 양상은 다음과 같다:

1. **조기 종료**: 에이전트가 작업 완료 전에 "충분히 좋다"고 자체 판단하여 작업을 중단한다. 남은 기능을 "선택적" 또는 "향후 추가"로 재분류한다.
2. **범위 축소**: 에이전트가 스펙에 명시된 기능 중 일부를 의도적으로 건너뛴다. "핵심 기능에 집중하겠다"는 표현으로 축소를 정당화한다.
3. **품질 기준 완화**: 에이전트가 자체적으로 품질 기준을 낮춘다. "프로토타입 수준"이라는 표현이 빈번하게 등장한다.

### 4.1.1 발생 메커니즘

Context anxiety의 근본 원인은 컨텍스트 누적이다. 에이전트가 작업을 수행하면서 생성되는 정보, 대화 이력, 디버깅 로그, 실패한 시도, 중간 결과물이 컨텍스트에 축적된다. 이 축적이 두 가지 문제를 야기한다.

첫째, **유효 컨텍스트의 축소**. 컨텍스트 윈도우의 총 용량은 고정되어 있으므로, 이력이 축적되면 새로운 정보를 수용할 공간이 줄어든다. 에이전트가 새로운 코드를 생성하거나 새로운 파일을 읽을 수 있는 여유가 감소한다.

둘째, **주의 분산**. 축적된 이전 정보가 현재 작업에 대한 에이전트의 주의를 분산시킨다. 이전 라운드에서 실패한 접근 방식, 디버깅 과정의 세부 사항, 폐기된 설계 결정이 컨텍스트에 남아 있으면, 에이전트는 이 정보를 참조하면서 인지적 부하가 증가한다.

### 4.1.2 하네스에서의 중요성

단일 에이전트 작업에서 context anxiety는 불편한 수준이지만, 하네스의 반복 루프에서는 치명적이다. 하네스는 3라운드 이상의 반복을 전제하며, 각 라운드에서 Generator가 새로운 빌드 또는 수정 빌드를 수행한다. 라운드가 진행될수록 컨텍스트에 이전 라운드의 평가 결과, 수정 이력, 디버깅 과정이 축적된다.

컨텍스트 축적이 통제되지 않으면, Round 2의 Generator는 Round 1의 모든 이력을 포함한 상태에서 작업해야 한다. Round 3에서는 Round 1과 2의 이력이 모두 포함된다. 이 누적은 기하급수적이 아니라 선형적이지만, 컨텍스트 윈도우의 유한성 때문에 3-5라운드 내에 한계에 도달한다.

## 4.2 Compaction과 Context Reset의 차이

컨텍스트 누적 문제에 대한 두 가지 접근이 있다. Compaction과 Context Reset이다. 이 두 접근은 표면적으로 유사하지만 본질적으로 다른 전략이다.

### 4.2.1 Compaction (요약 축소)

Compaction은 기존 컨텍스트를 요약하여 축소하는 접근이다. 전체 대화 이력을 압축된 요약으로 대체하여 컨텍스트 공간을 확보한다.

Compaction의 장점:
- 이전 컨텍스트의 연속성이 유지된다.
- 에이전트가 이전 작업의 맥락을 (요약 수준에서) 인식한다.

Compaction의 문제점:
- **정보 손실이 비선택적이다.** 요약 과정에서 어떤 정보가 보존되고 어떤 정보가 제거되는지를 제어하기 어렵다. 중요한 설계 결정이 요약에서 누락될 수 있고, 불필요한 디버깅 세부 사항이 요약에 포함될 수 있다.
- **오염 전파.** 이전 라운드의 잘못된 판단이나 실패한 접근이 요약에 포함되면, 다음 라운드의 에이전트에게 부정적 영향을 미친다. "이전에 X 접근을 시도했으나 실패했다"는 정보가 에이전트로 하여금 유효한 X 접근을 기피하게 만들 수 있다.
- **자기평가 편향의 전파.** Generator의 요약이 Evaluator에게 전달되면, clean-room 격리가 부분적으로 파괴된다.

### 4.2.2 Context Reset (완전 초기화 + 선택적 전달)

Context Reset은 각 라운드에서 에이전트를 새로운 컨텍스트로 시작하고, 이전 라운드의 정보를 **구조화된 handoff 파일**로만 전달하는 접근이다.

Context Reset의 장점:
- **정보 선택이 명시적이다.** Handoff 스키마에 정의된 필드만 전달되므로, 전달되는 정보와 차단되는 정보가 완전히 통제된다.
- **오염 차단.** 이전 라운드의 디버깅 이력, 실패한 시도, 내부 추론이 차단된다. 다음 라운드의 에이전트는 "무엇을 수정해야 하는가"만 알고, "이전에 무엇을 시도했는가"는 알지 못한다.
- **Clean-room 격리와 자연스러운 통합.** Evaluator의 clean-room 격리가 context reset의 부분 집합이다. 모든 에이전트가 독립된 컨텍스트에서 실행되므로, 격리가 아키텍처 수준에서 보장된다.

Context Reset의 비용:
- 각 라운드에서 에이전트가 프로젝트의 현재 상태를 처음부터 파악해야 한다. 코드 읽기, 파일 구조 탐색, 의존성 확인 등의 초기 오버헤드가 매 라운드 발생한다.
- 이 오버헤드는 Handoff의 풍부도에 반비례한다. Handoff가 충분한 맥락을 제공하면 에이전트의 초기 탐색 비용이 감소한다.

### 4.2.3 선택의 근거

하네스에서 Context Reset을 선택한 근거는 정보 통제의 명시성이다. Compaction에서 "어떤 정보가 보존되는가"는 요약 알고리즘에 의해 암묵적으로 결정된다. Context Reset에서는 Handoff 스키마에 의해 명시적으로 결정된다.

이 명시성은 하네스의 디버깅과 최적화에서 결정적 차이를 만든다. 라운드 간 정보 전달에 문제가 있을 때, Compaction에서는 "요약이 불충분했는가?"라는 모호한 질문에 직면하지만, Context Reset에서는 "Handoff 스키마의 어떤 필드가 누락되었는가?"라는 구체적 질문으로 진단할 수 있다.

## 4.3 Structured Handoff 파일 설계

### 4.3.1 설계 원칙

Handoff 파일은 Evaluator가 생성하여 Orchestrator를 거쳐 Generator에 전달되는 구조화된 문서다. 설계 시 다음 원칙을 적용한다.

**원칙 1: 고정 스키마.** Handoff의 구조는 라운드, 프로젝트, 실행과 무관하게 동일하다. "JSON or Markdown is not a design." 비정형 텍스트가 아닌 고정 필드의 고정 구조를 사용한다.

이 원칙의 근거는 다중 라운드 루프에서 비정형 출력이 야기하는 문제다. Evaluator가 자유 형식으로 피드백을 생성하면, Generator가 그 피드백을 파싱하는 과정에서 정보 손실이 발생한다. 고정 스키마는 파싱 없이 직접 참조할 수 있으므로 정보 전달의 신뢰성이 높다.

**원칙 2: 행동 지향.** Handoff의 모든 필드는 Generator의 구체적 행동으로 전환 가능해야 한다. "디자인이 좋지 않다"는 관찰이지 행동 지시가 아니다. "헤더의 글꼴 크기를 24px에서 32px로 변경하라"는 행동 지시다.

**원칙 3: 보존 항목 명시.** 수정해야 할 항목뿐 아니라, 수정하지 말아야 할 항목도 명시한다. 이는 회귀(regression) 방지를 위한 것이다. Generator가 특정 항목을 수정하면서 이미 합격한 항목을 파괴하는 것을 방지한다.

### 4.3.2 스키마 구조

Handoff 스키마의 필드 구성:

```
Handoff {
  round: number                    // 현재 라운드 번호
  verdict: "PASS" | "FAIL" | "PARTIAL"  // 평가 결과
  
  scores: {                        // 축별 점수 (1-10)
    design_quality: number
    originality: number
    craft: number
    functionality: number
  }
  
  thresholds_met: boolean          // 모든 임계값 충족 여부
  
  must_fix: [{                     // 반드시 수정해야 할 항목
    id: string                     // 고유 식별자
    description: string            // 수정 지시 (행동 지향적)
    severity: "critical" | "high" | "medium"
  }]
  
  preserve: [{                     // 수정 금지 항목 (회귀 방지)
    id: string
    description: string
  }]
  
  strategic_direction: "REFINE" | "PIVOT"  // 전략적 방향
  direction_rationale: string      // 방향 결정 근거
  
  cost_so_far: {                   // 비용 현황
    total_usd: number
    rounds_remaining: number
    budget_remaining_usd: number
  }
}
```

### 4.3.3 각 필드의 역할

**scores**: Generator에게 현재 상태의 정량적 지표를 제공한다. Generator는 점수가 낮은 축에 우선적으로 노력을 집중할 수 있다.

**must_fix**: Generator의 수정 작업 목록이다. severity에 따라 우선순위가 결정된다. critical 항목은 반드시 해결해야 하며, medium 항목은 예산이 허용하는 범위에서 해결한다.

**preserve**: 회귀 방지 목록이다. 이전 라운드에서 합격한 항목이 수정 과정에서 파괴되는 것을 방지한다. Generator는 preserve 목록의 항목을 변경하기 전에 그 변경이 불가피한지를 확인해야 한다.

**strategic_direction**: Generator의 수정 범위를 결정하는 이진 결정이다. REFINE은 현재 방향을 유지하며 세부 수정을 수행하라는 지시이고, PIVOT은 미학적 접근 전체를 재설계하라는 지시다. 이 결정은 Evaluator가 내리며, Generator가 재해석하지 않는다.

**cost_so_far**: Generator에게 비용 제약을 인식시킨다. 남은 예산과 라운드 수에 따라 Generator가 수정 범위를 자율적으로 조정할 수 있다.

## 4.4 Refine vs Pivot 결정 로직

### 4.4.1 두 전략의 차이

Evaluator가 내리는 strategic_direction 결정은 하네스의 핵심 분기점이다.

**REFINE**: 현재 접근의 방향성은 올바르나, 세부적 결함이 있다. 기존 코드와 디자인을 보존하면서 지적된 항목만 수정한다.

Generator의 REFINE 행동:
- must_fix 항목만 수정한다.
- preserve 항목은 변경하지 않는다.
- 디자인 언어(색상, 타이포그래피, 레이아웃 체계)를 유지한다.
- 수정 범위를 최소화한다.

**PIVOT**: 현재 접근의 방향성 자체에 문제가 있다. 세부 수정으로는 품질 향상이 어려우므로, 미학적 접근을 전체적으로 재설계한다.

Generator의 PIVOT 행동:
- 디자인 언어를 전면 재설계한다 (새로운 색상, 레이아웃, 타이포그래피).
- 기능적 preserve 항목은 유지한다 (기능 구현은 보존, 시각적 표현만 변경).
- 수정 범위가 REFINE보다 넓다.

### 4.4.2 결정 기준

PDF에서 PIVOT 결정의 근거를 다음과 같이 기술한다:

> "I also instructed the generator to make a strategic decision after each evaluation: refine the current direction if scores were trending well, or pivot to an entirely different aesthetic if the approach wasn't working."

결정 기준을 형식화하면:

- **REFINE 조건**: design_quality와 originality 점수가 임계값에 근접하거나 상승 추세이며, 지적된 결함이 세부적(특정 컴포넌트, 특정 색상, 특정 간격)인 경우.
- **PIVOT 조건**: design_quality 또는 originality 점수가 임계값보다 현저히 낮고, 지적된 결함이 전체적(전반적 미학, 레이아웃 체계, 색상 체계)인 경우.

### 4.4.3 비용과의 관계

PIVOT은 REFINE보다 비용이 높다. 디자인 전면 재설계는 더 많은 파일 수정을 수반하기 때문이다. 그러나 세부 수정을 반복해도 방향성 자체가 잘못되었다면 점수가 정체(plateau)할 뿐 개선되지 않는다. 이 경우 PIVOT의 높은 비용이 점수 정체의 무한 반복보다 비용 효율적이다.

Evaluator는 이 비용 트레이드오프를 고려하여 direction_rationale에 결정 근거를 기술한다. "색상 체계의 근본적 문제로 세부 수정 3회 반복 대비 전면 재설계가 비용 효율적"이라는 식의 근거 제시가 기대되는 형식이다.

## 4.5 고정 스키마의 중요성

Handoff 스키마 설계에서 "고정 스키마"를 강조하는 이유를 별도로 논의할 필요가 있다.

### 4.5.1 비정형 출력의 문제

에이전트에게 "피드백을 자유 형식으로 작성하라"고 지시하면, 라운드마다 다른 구조의 출력이 생성된다. Round 1에서는 글머리 기호 목록을, Round 2에서는 문단 형식을, Round 3에서는 표를 사용할 수 있다.

이 비일관성이 다중 라운드 루프에서 문제를 야기하는 메커니즘:

1. Generator가 피드백의 구조를 먼저 파악해야 하므로, 실제 수정 작업에 투입되는 컨텍스트가 감소한다.
2. 구조가 다르면 동일한 의미의 피드백이 다른 방식으로 해석될 수 있다.
3. Orchestrator가 피드백을 프로그래밍 방식으로 처리(예: must_fix 항목 수 확인, 점수 비교)하는 것이 불가능하거나 불안정해진다.

### 4.5.2 스키마 검증의 가치

고정 스키마는 Zod와 같은 런타임 검증 라이브러리로 검증할 수 있다. Evaluator의 출력이 스키마를 만족하지 않으면 파싱 단계에서 즉시 거부된다. 이 조기 실패(fail-fast)가 잘못된 피드백이 Generator에 전달되는 것을 방지한다.

스키마 검증이 포착하는 오류 유형:
- 필수 필드 누락 (must_fix가 빈 배열일 때 verdict가 FAIL인 경우)
- 값 범위 위반 (점수가 1-10 범위를 벗어난 경우)
- 타입 불일치 (severity가 "critical"/"high"/"medium" 외의 값인 경우)
- 구조 불일치 (예상치 못한 중첩 구조)

스키마 검증 실패 시의 복구 전략은 Chapter 7에서 상세히 다룬다.
