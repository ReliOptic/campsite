# Chapter 3: 자기평가 편향과 Clean-Room 격리

## 3.1 PDF의 핵심 발견

PDF 저자가 하네스 개발 과정에서 가장 먼저 마주한 구조적 문제는 자기평가 편향이었다. 이 문제의 심각성을 PDF 원문이 직접 기술한다:

> "Out of the box, Claude is a poor QA agent. In early runs, I watched it identify legitimate issues, then talk itself into deciding they weren't a big deal and approve the work anyway."

이 관찰에서 주목할 점은 "identify legitimate issues" 부분이다. 모델은 결함을 **인식**한다. 문제는 인식 능력이 아니라 **판단 과정**에 있다. 결함을 인식한 후, 그 결함의 심각도를 하향 조정하고, 최종적으로 합격 판정을 내리는 자기합리화 과정이 체계적으로 발생한다.

이 현상은 특정 모델의 결함이 아니다. 동일 컨텍스트에서 생성과 평가를 수행하는 모든 시스템에서 발생할 수 있는 구조적 문제다.

## 3.2 자기합리화 패턴 분류

PDF의 관찰과 하네스 설계 과정에서 도출된 자기합리화 패턴을 4가지로 분류한다.

### 3.2.1 코드 독해 대체 (Code Reading Substitution)

**패턴**: "코드를 읽어본 바로는 정확해 보인다."

에이전트가 실제 실행 없이 코드를 읽는 것으로 검증을 대체하는 패턴이다. 코드의 논리적 정합성은 코드 독해로 일부 검증 가능하지만, 런타임 동작, 경계 조건, 사용자 상호작용의 정확성은 실행 없이는 검증 불가능하다.

이 패턴이 위험한 이유는 에이전트가 코드를 작성한 당사자이기 때문이다. 작성자가 자신의 코드를 읽을 때, 코드의 의도를 이미 알고 있으므로 의도와 구현의 불일치를 간과하기 쉽다. 소프트웨어 공학에서 코드 리뷰를 작성자 외의 인원이 수행하는 이유와 동일한 원리다.

### 3.2.2 심각도 하향 (Severity Downgrade)

**패턴**: "사소한 문제이므로 통과시키겠다."

에이전트가 기준 미달인 결함을 발견한 후, 그 결함을 "사소하다"고 재분류하여 합격 판정을 유지하는 패턴이다. 이 재분류에는 객관적 근거가 없다. 에이전트가 자체적으로 심각도 기준을 완화한 것이다.

이 패턴은 특히 디자인 품질 평가에서 빈번하게 발생한다. "버튼 색상이 배경과 대비가 부족하지만 전체적인 디자인 방향은 올바르다"는 식의 합리화가 전형적이다. 대비 부족은 접근성 기준 미달이며 "사소한 문제"가 아님에도, 전체적 맥락을 원용하여 심각도를 하향한다.

### 3.2.3 총괄적 승인 (Holistic Approval)

**패턴**: "전반적으로 좋은 품질이다."

개별 기준에서 미달인 항목이 있음에도, "전반적" 또는 "종합적"이라는 표현으로 총괄 승인을 내리는 패턴이다. 이 패턴의 메커니즘은 높은 점수를 받은 축이 낮은 점수를 받은 축을 상쇄하는 것이다.

예를 들어 functionality가 9점이고 design_quality가 4점인 경우, "기능적으로 우수하므로 전반적으로 좋은 품질"이라고 판단한다. 그러나 design_quality 4점은 임계값(7점) 미달이며, 이는 단독으로 FAIL 사유에 해당한다.

이 패턴을 방지하기 위해 하네스는 총점 평균이 아닌 **개별 축의 hard threshold**를 적용한다. 4개 축 중 어느 하나라도 임계값 미만이면 verdict는 FAIL이다.

### 3.2.4 구현자 신뢰 (Implementer Trust)

**패턴**: "구현자의 테스트가 이미 통과한다."

에이전트가 자신이 작성한 테스트의 통과를 독립적 검증으로 간주하는 패턴이다. 자신이 작성한 코드를 검증하기 위해 자신이 작성한 테스트를 사용하는 것은 순환 논증(circular reasoning)이다. 테스트가 코드와 동일한 오해를 반영할 수 있기 때문이다.

이 패턴은 특히 경계 조건 테스트에서 문제가 된다. 에이전트가 경계 조건을 인식하지 못한 채 코드를 작성했다면, 그 에이전트가 작성한 테스트에도 동일한 경계 조건이 누락될 확률이 높다.

## 3.3 Clean-Room 해법

### 3.3.1 격리 원칙

Clean-room 격리는 Evaluator의 입력을 엄격히 제한하여 자기합리화의 원재료를 구조적으로 제거하는 접근이다.

자기합리화가 발생하려면 에이전트가 Generator의 **의도**에 접근할 수 있어야 한다. "이 코드는 X를 의도했으므로 결과가 X에 부합한다"는 합리화는 Generator의 의도를 알아야만 가능하다. Generator의 의도를 차단하면, Evaluator는 산출물 자체만으로 판단해야 한다.

### 3.3.2 수신 정보와 차단 정보

Evaluator가 수신하는 정보:

| 정보 유형 | 출처 | 용도 |
|----------|------|------|
| ProductSpec | Planner | 평가 기준의 근거 |
| AcceptanceContract | Orchestrator | 합격 기준 목록 |
| 생성된 코드 | 파일 시스템 | 정적 분석 대상 |
| 실행 중인 앱 | Playwright | 동적 검증 대상 |
| 평가 기준 | evaluator-system.md | 채점 프레임워크 |

Evaluator가 수신하지 않는 정보:

| 차단 정보 | 차단 이유 |
|----------|----------|
| Generator의 시스템 프롬프트 | Generator의 설계 의도를 알면 의도에 동조하는 합리화 발생 |
| Generator의 의사결정 로그 | "이 선택에는 이유가 있다"는 합리화의 원재료 |
| Generator의 디버깅 과정 | "이미 충분히 시도했다"는 동정적 판단 유발 |
| 이전 라운드의 Generator 컨텍스트 | 누적된 컨텍스트가 판단 독립성을 오염 |
| Planner의 내부 추론 | 스펙의 "의도"가 아닌 스펙의 "문면"으로만 판단해야 함 |

### 3.3.3 구현 메커니즘

Agent SDK에서 clean-room 격리는 자연스럽게 구현된다. 각 agent() 호출은 독립적인 컨텍스트로 시작하기 때문이다. Evaluator를 별도의 agent() 호출로 실행하면, Generator의 컨텍스트는 자동으로 차단된다.

구현 시 핵심 함수는 `buildCleanRoomPrompt()`이다. 이 함수는 Evaluator에게 전달할 사용자 프롬프트를 구성하되, 위 표의 "수신하는 정보"만을 포함한다:

```
buildCleanRoomPrompt(spec, contract, thresholds) → string

구성:
  1. "## Product Spec" + spec의 JSON 직렬화
  2. "## Acceptance Contract" + contract의 JSON 직렬화
  3. "## Score Thresholds" + 축별 임계값 명시
  4. "Any criterion below its threshold means FAIL."
```

이 함수의 설계에서 의도적인 선택이 두 가지 있다. 첫째, 입력을 고정 형식으로 구성하여 누락이나 추가를 방지한다. 둘째, 마지막 줄에 "Any criterion below its threshold means FAIL"을 명시하여 총괄적 승인 패턴을 사전에 차단한다.

## 3.4 Anti-Rationalization 프롬프트 설계

Clean-room 격리가 자기합리화의 **원재료**를 차단한다면, anti-rationalization 프롬프트는 자기합리화의 **행동 패턴**을 명시적으로 금지한다. 두 접근은 상호 보완적이다.

### 3.4.1 패턴 명시와 거부 지시

Evaluator의 시스템 프롬프트에 자기합리화 패턴을 구체적으로 열거하고, 각 패턴에 대한 올바른 대응을 지시한다:

| 자기합리화 패턴 | 올바른 대응 |
|---------------|-----------|
| "코드를 읽어본 바로는 정확해 보인다" | 실제로 실행하라 |
| "구현자의 테스트가 이미 통과한다" | 독립적으로 테스트하라 |
| "사소한 문제이므로 통과시키겠다" | 사소해도 기준 미달이면 FAIL |
| "전반적으로 좋은 품질이다" | 기준별로 개별 채점하라 |

이 접근의 효과는 패턴을 명명(naming)하는 데 있다. 모델은 명명된 패턴을 인식하고 회피하는 능력이 뛰어나다. "자기합리화를 하지 마라"라는 추상적 지시보다, "이러이러한 패턴이 자기합리화다"라는 구체적 열거가 훨씬 효과적이다.

### 3.4.2 판단 순서 강제

Anti-rationalization 프롬프트의 두 번째 전략은 판단 순서를 강제하는 것이다. Evaluator가 먼저 verdict를 결정하고 나중에 근거를 제시하면, 원하는 verdict에 맞춰 근거를 선택적으로 구성할 위험이 있다.

이를 방지하기 위해 프롬프트에서 다음 순서를 강제한다:

1. 각 축별로 구체적 관찰 사항을 먼저 기술한다.
2. 관찰에 기반하여 축별 점수를 채점한다.
3. 축별 점수와 임계값을 비교한다.
4. 비교 결과에 기반하여 verdict를 결정한다.

이 순서에서 verdict는 과정의 **결과**이지 **전제**가 아니다. "PASS를 주고 싶으니 근거를 찾겠다"가 아니라 "근거를 분석했더니 PASS/FAIL이 결정되었다"의 흐름을 강제한다.

### 3.4.3 부정적 발견 우선 지시

인간 평가자에게도 적용되는 원칙이지만, 모델에게 특히 유효한 전략이다. Evaluator에게 "먼저 결함을 찾고, 그 다음에 긍정적 측면을 평가하라"고 지시한다.

이 순서가 중요한 이유는 발견의 순서가 판단에 영향을 미치기 때문이다. 긍정적 측면을 먼저 열거하면, 이후에 발견된 결함의 심각도를 무의식적으로 하향 조정하는 경향이 있다. "이렇게 좋은 점이 많으니 이 결함은 사소하겠지"라는 합리화가 자연스럽게 따라온다. 결함을 먼저 열거하면 이 효과가 약화된다.

## 3.5 Few-Shot 캘리브레이션

Anti-rationalization 프롬프트가 "무엇을 하지 말라"를 지시한다면, few-shot 캘리브레이션은 "어떻게 해야 하는지"를 시연한다.

### 3.5.1 캘리브레이션의 목적

모델마다, 그리고 동일 모델이라도 프롬프트에 따라 점수 분포가 다르다. 어떤 모델은 7-9 범위에 점수가 집중되고, 다른 모델은 3-7 범위에 분포한다. 캘리브레이션은 Evaluator의 채점 척도를 의도한 분포에 맞추는 작업이다.

### 3.5.2 캘리브레이션 예시 구성

효과적인 캘리브레이션 예시는 다음 조건을 만족해야 한다:

1. **점수 범위 전체를 포함**: 3점 사례, 5점 사례, 7점 사례, 9점 사례를 각각 포함하여 척도의 전 구간에 앵커를 설정한다.
2. **경계 사례를 포함**: 임계값 직상(7점)과 직하(6점)의 사례를 포함하여, 경계에서의 판단 기준을 명확히 한다.
3. **verdict와 점수의 관계를 시연**: 개별 축의 점수가 임계값 미달일 때 verdict가 FAIL이 되는 사례를 포함한다.

### 3.5.3 캘리브레이션의 한계

Few-shot 캘리브레이션에는 본질적 한계가 있다. 예시가 다루지 않는 상황에서의 판단은 여전히 모델의 자체 기준에 의존한다. 또한 예시가 과도하면 Evaluator가 예시와의 유사성에 기반하여 판단하는 "패턴 매칭"에 빠질 위험이 있다.

이 한계를 인식하되, clean-room 격리 + anti-rationalization 프롬프트 + few-shot 캘리브레이션의 3중 방어가 어느 단일 전략보다 효과적이라는 점이 실무적 결론이다. 각 전략이 다른 전략의 한계를 보완한다.

## 3.6 자기평가 편향과 모델 진화

PDF V1→V2 진화에서 흥미로운 관찰은 모델 능력의 향상(Opus 4.5 → 4.6)이 자기평가 편향을 **완화**하지만 **해소**하지는 않는다는 점이다. 더 강력한 모델은 자기합리화 패턴을 인식하는 능력이 향상되지만, 동일 컨텍스트에서 생성과 평가를 수행하는 구조적 문제는 모델 능력과 무관하게 존재한다.

이것이 Chapter 5에서 다루는 설계 원칙 1("모든 구성요소는 모델 능력에 대한 가정을 인코딩한다")의 예외적 적용 사례다. Clean-room Evaluator는 모델 능력 향상으로 불필요해질 가능성이 낮은 구성요소다. 자기평가 편향은 모델의 능력 부족이 아닌 아키텍처의 구조적 속성이기 때문이다.

그러나 anti-rationalization 프롬프트의 구체적 패턴 목록은 모델 진화에 따라 갱신이 필요할 수 있다. 새로운 모델이 기존 패턴을 학습하여 회피하면서 새로운 형태의 합리화를 생성할 수 있기 때문이다. 이 갱신 필요성은 하네스 운영의 지속적 과제다.
