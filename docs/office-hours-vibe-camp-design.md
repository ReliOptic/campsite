# Design: Campsite as a Vibe Coding Camp

Generated by /office-hours on 2026-04-03

Status: DRAFT
Mode: Builder

## Problem Statement

Campsite should become the place a solo vibe coder returns to when their head is full of half-finished threads.

The problem is not just "too many agents." The real problem is recovery time. A builder leaves to do their day job, check GitHub, read research, or work on another project, then comes back and spends too long asking:

- What was I making?
- Which agent is still running?
- Which result matters?
- What should I do next?

That re-entry cost kills momentum, pushes work later into the night, and breaks the feeling of deep creative progress.

## What Makes This Cool

- Multiple agents and terminals do not feel like scattered processes. They feel like one shared camp.
- The UI is a local 2D lofi pixel-art space, not a gray dashboard.
- The camp visual language makes state legible without feeling corporate.
- The point is not automation theater. The point is preserving good creative energy while real work keeps moving.
- A user can step away, come back, and understand the whole camp in about five seconds.
- A user can work in the terminal, slip into Campsite to recover and reorient, then slip back into execution without feeling like they changed products.

## Constraints

- This should start with a solo builder as the wedge, not a full multiplayer collaboration system.
- The product must help people re-enter deep work, not create more noise.
- Gamification must clarify state, not hide it.
- Auto-running more tasks after completion is dangerous if it burns tokens or causes silent drift.
- The system should prefer leaving behind useful state over inventing more work.
- The experience should still feel at home in a terminal-first workflow.

## Premises

- Vibe coding retention matters more than raw task throughput for the first version.
- Recovery friction is a sharper problem than orchestration complexity.
- A spatial metaphor can outperform a plain task list because it helps humans rebuild mental context faster.
- "Done" is often the wrong state label for creative and engineering work. Work usually changes phase rather than ending.
- Agent autonomy should be bounded. The safest automation is status capture, not unbounded follow-on execution.
- The right first experience is not "look at all the agents" but "I came back and instantly know what is going on."

## Approaches Considered

### Approach A: Agent Control Tower

Build Campsite like a control dashboard for multiple agents:

- Lists of running agents
- Logs, statuses, token counts
- Manual orchestration controls
- Emphasis on concurrency and automation

Pros:

- Easy to explain
- Familiar to power users
- Directly addresses multi-agent sprawl

Cons:

- Feels like infrastructure, not a place
- Risks becoming another observability screen
- Solves visibility, but not emotional re-entry
- Weak differentiation because many orchestration tools already live here

### Approach B: Recovery-First Camp Space

Build Campsite as a local camp that visualizes the state of work in a way that supports re-entry:

- Agents "enter the camp" from Claude, Codex, Gemini, Ghostty, or other local terminals
- Fire-state metaphors show phase and required human involvement
- The return moment is the hero interaction
- The campfire becomes the central mission thread

Pros:

- Clear product identity
- Better emotional hook for solo builders
- Stronger fit with the stated mission of helping people make new things
- More defensible than plain orchestration because it changes how work feels, not just how it is scheduled

Cons:

- Harder to design well
- Easy to become cute but shallow
- Requires discipline so game language stays useful

## Recommended Approach

Choose Approach B.

Campsite should be a recovery-first work space for solo vibe coders. The product's magic moment is not "many agents run at once." It is "I came back after context switching and instantly understood my world."

That means the first product loop should be:

- leave the camp
- agents continue or settle into a visible state
- return later
- read the camp in five seconds
- re-enter deep work immediately

This is a stronger wedge than team collaboration and a stronger identity than generic orchestration.

## Open Questions

- What is the exact first-launch UX for getting terminals and agents into the camp?
- How much state can be inferred automatically versus explicitly reported?
- How should "campfire" be modeled, one mission at a time or multiple concurrent fires?
- What is the minimum useful pixel-art scene before visual polish becomes a distraction?
- Which actions are safe to automate after an agent reaches a resting state?
- How should this world later extend to team camps, shared conversation, branch handoff, and person-to-person mission transfer?

## Success Criteria

- A solo user can leave and return to a project and regain situational awareness in 5 seconds.
- The user can tell three things at a glance:
- who is still working
- what changed or reached a review point
- what the next human action should be
- The product feels fun without obscuring task state.
- Users report lower restart friction after interruptions.
- The first demo makes people say "I want to try this," not just "nice dashboard."

## Distribution Plan

- Deliver as a local-first tool that fits terminal-driven workflows.
- Keep the entry point simple: launch or enter Campsite from the terminal and open a local web UI for the spatial view.
- Treat the web scene as the visible camp and the terminal as the working edge.
- Existing release mechanics for the CLI can carry the first version; no separate hosted service should be required.

## Next Steps

- Define the recovery loop in one sentence and use it as the product test for every feature.
- Design the first five-second return screen around exactly three signals:
- who is active
- what reached a human review point
- what the next action is
- Implement the fire-state model:
- `bulssi` for newly started work that needs watching
- `modakbul` for active focused execution
- `deungbul` for a completed segment waiting on human judgment
- `yeongi` for blocked work that needs help
- `jangjak` for the next prepared action
- Sketch the smallest pixel-art camp scene that can encode those states clearly.
- Delay multiplayer and team camp features until the solo recovery loop feels strong.
- Add bounded automation later through explicit policies, for example "after this finishes, run tests," rather than free-running autonomy.
